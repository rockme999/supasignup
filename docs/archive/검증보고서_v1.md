# 번개가입 - 문서 교차 검증 보고서

**검증 대상**: PRD v1.2, 서비스계획서 v1.1, 기술스펙 v1.1
**검증일**: 2026-03-12
**최종 업데이트**: 2026-03-12 (결정 사항 반영)

## 1. 심각도 높음 - MVP 전에 반드시 해결

### [H1] OAuth 플로우 아키텍처 모순 (가장 중요)

**문제**: 위젯이 직접 `/oauth/authorize`를 호출하는 플로우와, 카페24 SSO가 `/oauth/authorize`를 호출하는 플로우가 혼재되어 있음.

**상세**: 카페24 SSO 표준 플로우는:
```
사용자 → 카페24 SSO 버튼 클릭 → 카페24가 번개가입 authorize URL 호출 → 번개가입 인증 → 카페24로 복귀
```

그런데 PRD에서는 위젯이 직접 authorize URL을 호출:
```
사용자 → 위젯의 소셜 버튼 클릭 → 위젯이 번개가입 authorize URL 직접 호출 → ???
```

**위젯이 직접 호출하면 카페24 SSO 플로우가 깨집니다.** 카페24가 Authorization Code를 받아 회원 생성/세션 생성을 하려면, 반드시 카페24가 플로우의 시작점이어야 합니다.

**해결안 2가지:**

**(A안) 위젯이 카페24 SSO를 트리거하는 방식**
```
위젯 "구글로 가입" 버튼 클릭
  → 카페24 SSO 로그인 URL 호출 (provider 파라미터 포함)
  → 카페24가 번개가입 authorize URL 호출 (카페24가 state 생성)
  → 번개가입이 provider 파라미터를 보고 바로 Google OAuth로 리다이렉트
  → Google 인증 → 번개가입 콜백 → 카페24 콜백 → 회원생성+로그인
```
- 장점: SSO 표준 플로우 유지
- 단점: 카페24 SSO URL에 provider를 전달하는 방법 확인 필요

**(B안) 위젯이 직접 호출하고, 카페24 SSO를 우회하는 방식**
```
위젯 "구글로 가입" 버튼 클릭
  → 번개가입 authorize URL 직접 호출
  → Google 인증 → 번개가입 콜백
  → 번개가입이 카페24 Admin API로 직접 회원 생성
  → 번개가입이 프론트에서 카페24 로그인을 트리거 (SSO 경유)
```
- 장점: 1-클릭 유지 가능
- 단점: SSO를 우회하므로 로그인 세션 생성이 복잡

--> ✅ **해결됨** (v1.2): 카페24 SSO에 provider별 4개 SSO 항목을 별도 등록하는 방식으로 결정. 각 SSO 항목의 Authorize URL에 `?provider=google` 등 포함. 카페24가 플로우 시작점. Phase 2 추가 provider는 ScriptTag 위젯 방식 사용 (SSO 5개 제한).

### [H2] redirect_uri 검증 부재 (Open Redirect)

**문제**: `/oauth/authorize`에서 받은 `redirect_uri`를 검증 없이 사용하여 Open Redirect 공격 가능.

**해결**: shops 테이블에 `allowed_redirect_uris` 필드 추가, authorize 요청 시 화이트리스트 대조.

--> ✅ **반영 완료** (기술스펙 v1.1): shops 테이블에 `allowed_redirect_uris` 추가, authorize 요청 시 화이트리스트 대조 검증 로직 명세.

### [H3] 소셜 OAuth에 PKCE 미적용

**문제**: Authorization Code 탈취 방지를 위한 PKCE가 설계에 없음.

**해결**: 소셜 OAuth 요청 시 `code_verifier`/`code_challenge` 생성 로직 추가.

--> ✅ **반영 완료** (기술스펙 v1.1): 소셜 OAuth 요청 시 PKCE code_verifier/code_challenge 생성, KV에 저장, token 교환 시 검증 로직 명세.

### [H4] 개인정보 평문 저장

**문제**: `users.email`, `users.name`, `users.raw_data`가 D1에 평문 저장. `raw_data`에는 전화번호, 생일 등도 포함될 수 있음.

**해결**: 개인정보 필드 AES-GCM 암호화 적용. 검색이 필요한 필드는 해시 인덱스 별도 저장.

--> ✅ **반영 완료** (기술스펙 v1.1): email, name, raw_data AES-GCM 암호화. email_hash (SHA-256) 인덱스 별도 저장.

### [H5] member_count 조작 가능성

**문제 1**: 대시보드 API에서 `shops` 수정 시 `member_count` 직접 리셋 가능
**문제 2**: 쇼핑몰 삭제 후 재등록하면 count 초기화
**문제 3**: 재로그인 시에도 count 증가 가능 (가입/로그인 미구분)

**해결**:
- API에서 member_count 필드 수정 명시적 차단
- 삭제 시 이전 count 보관 (soft delete)
- shop_users 존재 여부로 가입/재로그인 구분

--> ✅ **반영 완료** (기술스펙 v1.1): member_count 폐지 → shop_users 월별 count로 계산. API에서 직접 수정 차단, soft delete, shop_users 존재 여부로 가입/재로그인 구분.

### [H6] 카페24 앱 설치 플로우 누락

**문제**: 카페24 앱스토어에서 앱 설치 → OAuth 인증 → platform_access_token 발급 → ScriptTag 자동 등록의 전체 플로우가 기술스펙에 없음.

**해결**: 앱 설치 플로우 상세 설계 추가 필요.

--> ✅ **반영 완료** (기술스펙 v1.1, 서비스계획서 v1.1): 앱 설치 플로우 상세 설계 추가 (GET /api/cafe24/install, /api/cafe24/callback). 서비스계획서에 시나리오 A-0 추가.

### [H7] 과금 단위 불명확

**문제**: 무료 100명이 쇼핑몰별인지 운영자 전체 합산인지 불명확.

--> ✅ **해결됨** (v1.2): 쇼핑몰당 매월 신규 가입 100명. 한도 초과 시 버튼 비활성화/숨김. 기존 회원은 카페24 자체 로그인으로 유지.

### [H8] 개인정보 처리방침 미작성

**문제**: 법적 필수 사항인 개인정보 처리방침이 없음.

--> ⏳ **출시 전 작성 필요** (PRD v1.2에 필수 사항으로 명시)

## 2. 심각도 중간 - 출시 전 권장

### [M1] 누락 시나리오: 앱 해제/탈퇴

운영자가 번개가입 사용을 중단할 때:
- ScriptTag 제거
- D1 데이터 보관/삭제 정책
- 기존 소셜 가입 회원의 로그인 불가 문제 (카페24 SSO가 끊어지므로)

--> ✅ **반영 완료** (서비스계획서 v1.1): 시나리오 H 추가 - ScriptTag 제거, 30일 데이터 보관, 기존 회원 카페24 자체 로그인 안내.

### [M2] 누락 시나리오: 유료 → 무료 다운그레이드

구독 만료 시 100명 초과 기존 회원들의 로그인은?
- 신규 가입만 차단하고 기존 회원 로그인은 유지가 합리적

--> ✅ **해결됨**: 매월 신규 가입 100명 기준이므로 다운그레이드 시 당월 신규 가입만 제한. 기존 회원 로그인은 카페24 자체 로그인으로 유지.

### [M3] 누락 시나리오: 여러 소셜로 같은 쇼핑몰 중복 가입

사용자가 Google로 가입 후 카카오로 다시 가입하면 카페24에 2명의 회원 생성됨.

--> (카페24 이메일 중복체크 설정으로 일부 방어 가능, 시나리오 정리 필요)

### [M4] 누락 시나리오: 비밀번호 찾기

대시보드 운영자 비밀번호 분실 시 재설정 플로우 없음.

--> (API 추가 필요)

### [M5] 대시보드 brute force 방지

로그인 API에 rate limiting, 계정 잠금 정책 없음.

--> ✅ **반영 완료** (기술스펙 v1.1): KV 기반 IP별 시도 횟수 추적. 5회 실패 시 5분 잠금.

### [M6] 위젯 XSS 방어

`innerHTML` 직접 사용은 XSS 위험. `textContent` + `createElement` 사용 필요.

--> ✅ **반영 완료** (기술스펙 v1.1): innerHTML 대신 createElement + textContent 방식으로 변경.

### [M7] state 파라미터 분리

카페24 state와 소셜 OAuth state를 동일하게 사용하는 것은 부적절.

--> ✅ **반영 완료** (기술스펙 v1.1): cafe24_state와 social_state 분리. KV oauth_session에 별도 저장.

### [M8] 모바일 앱 내 웹뷰 OAuth

카카오/네이버 앱 내 웹뷰에서 OAuth 리다이렉트 시 쿠키/팝업 제한 문제.

--> (테스트 후 대응)

### [M9] 결제 수단 미결정

유료 전환 시 실제 결제를 어떻게 받을지 미결정.

--> ✅ **해결됨**: 카페24 수동 인앱결제 (모든 플랫폼 동일). automatic_payment="T" 별도 문의 예정.

### [M10] Account Linking (카페24 SSO 동일 이메일 처리)

**조사 결과**: 카페24가 "SNS 가입 시 동일 이메일 계정 연동" 기능을 내장하고 있음.
- 설정: 관리자 > 고객 설정 > SNS 로그인 연동 관리 > "SNS 가입 시 동일 이메일 연동" 활성화
- 동작: 동일 이메일 기존 회원 발견 → 연동 화면 → 기존 계정 ID/PW 입력 → 연동 완료
- **우리 역할**: SSO 설정 가이드에서 이 옵션 활성화를 안내

--> ✅ **반영 완료** (PRD v1.2, 서비스계획서 v1.1, 기술스펙 v1.1): SSO 설정 가이드에 활성화 안내 포함.
