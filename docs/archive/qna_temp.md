SupaSignup - 핵심 컨셉 정리                                                                                                                                                       
                                                                                                                                                                                  
  제가 이해한 내용을 정리하면:

  문제

  - 쇼핑몰 회원가입 과정이 복잡하면 이탈률이 높음
  - 각 쇼핑몰 플랫폼(카페24, 아임웹, 고도몰, 샵바이)마다 OAuth 연동을 개별 구현해야 하는 부담

  솔루션

  - 원클릭 소셜 가입: 이름/이메일 같은 최소 정보만으로 가입 완료
  - 점진적 프로필 수집: 주문 시점에 배송지, 연락처 등 상세정보 입력 (Progressive Profiling)
  - 간편 연동: 쇼핑몰에 스크립트 코드 한 줄 삽입으로 작동
  - 멀티 플랫폼: 카페24, 아임웹, 고도몰, 샵바이, 자체몰 지원

--> 일단은 대부분의 사용자가 자기가 주로 사용하는 계정은 이미 로그인 되어 있을 확률이 높다. 예를 들면 안드로이드 사용자라면 구글계정이이 있고 바로 사용할 수 있을 것이고 피시 사용자도 구글이나 카카오 또는 네이버에 이미 계정을 가지고 있거나 로그인까지 되어 있을 확률이 높다. 우리는 회원 가입단에 구글로가입, 애플로가입, 네이버로가입, 인스타로가입 등 간단한 버튼을 만들어 놓고 사용자가 해당 버튼을 클릭했을때 이미 로그인 되어 있다면 oauth로 넘어오는 정보로 일단 우리 디비에 정보를 입력하고 해당 쇼핑몰 임시계정을 생성한다. 향후에 주문 단계에 이르러서야 추가로 필요한 정보를 입력하게 해서 우리 디비와 쇼핑몰 디비에 업데이트 한다. 우리 디비에 임시계정을 생성하는 이유는 우선 쇼핑몰에 임의의 사용자 계정을 만들었다가 나중에 정보가 업데이트 될때 해당 쇼핑몰에 회원 정보를 업데이트 해 주기 위한 임시 저장이지만 만일 이 프로세스가 필요 없다면 우리는 디비를 아예 유지 하지 않아도 된다. 중요한 것은 oauth로 이름이나 이메일 또는 전화번호 아무것도 넘어오는것이 없더라도 뭔가 유니크한 해당 사용자의 데이타가 올것이므로 그것으로 일단 임시 회원을 생성해서 해당 쇼핑몰을 거의 회원가입 절차 없이 이용하게 하자는것이 제일 중요한 프로젝트의 목적이다. 



  지원 OAuth 프로바이더

  Apple, Google, Kakao, Naver, Discord, Telegram, Facebook, Instagram + 핸드폰번호, 이메일

  몇 가지 의견을 나누고 싶습니다:

  1. 비즈니스 모델은 어떻게 생각하시나요?
  - SaaS (월 구독) vs 가입 건수당 과금 vs 프리미엄 모델?  
  --> 그냥 무료로 회원가입 100명까지(나중에 변경될수 있음) 그 다음은 유료 월29900원(이것도 나중에 변경될 수 있음)과 연간 329900원

  2. 가입 UI 형태
  - 팝업 모달 (쇼핑몰 페이지 위에 뜨는 형태)?
  - 임베디드 위젯 (페이지에 직접 삽입)?
  - 리다이렉트 방식 (SupaSignup 페이지로 이동 후 돌아오기)?
  - 저는 팝업 모달이 사용자 경험상 가장 좋을 것 같습니다.
  --> 팝업 모달은 안되고 쇼핑몰의 회원가입 페이지 아래에 버튼처럼 달아서 이미 로그인된 oauth라면 바로 가입되게

  3. 데이터 소유권
  - 회원 데이터는 쇼핑몰 DB에 직접 저장? 아니면 SupaSignup 중앙 서버에서 관리하고 API로 제공?
  - 중앙 관리 시: 하나의 계정으로 여러 쇼핑몰 로그인 가능 (통합 ID)
  - 쇼핑몰 DB 저장 시: 쇼핑몰 자체 시스템과 호환성 좋음
  --> 각 oauth들이 반환하는 정보가 다 다른데도 카페24등의 플랫폼에서 제공하는 회원정보로 처리가 가능하다면 그렇게 하고 아니면 우리 디비도 임시 저장에 사용해서 처리 이것은 다른 플랫폼들의 회원 정보 쓰기 기능에 어떤 것들이 있는지 확인하고 기획해야 할거 같다.

  4. Progressive Profiling 시점
  - 첫 주문 시 배송 정보 입력
  - 그 외에 추가 수집이 필요한 시점이 있을까요? (예: 쿠폰 발급 시, 마케팅 수신 동의 등)
  --> 하기의 경쟁업체의 서비스와 비교해서 좀 더 고민 해봐야 할거 같은데....우리는 어떻게 할까? 회원가입 부분 이외에 추가 서비스를 포함시킬까? 의견을 나눠보자

  5. 기술 스택 선호도
  - 백엔드: Supabase (프로젝트명으로 보아) + Edge Functions?
  - 프론트: 임베디드 위젯용 Vanilla JS SDK?
  - 관리자 대시보드 필요 여부?
  --> 이건 디비를 써야 한다면 우리 로컬에 서버를 두고 클라우드 플래어 터널로 서비스 하던가 아니면 디비없이 간단하게 서비스 가능하다면 클라우드플레어 워커로 처리 할 수 있게 구현하면 가장 좋고...
  일단 수파베이스는 절대 안쓸것임, 클라우드에 RDBMS가 필요하다면 클라우드플레어또는 AWS등 다른 서비스를 고려할 것인데 서비스가 활성화 되기 전까지는 RDBMS가 필요하면 터널로 로컬에 서버구축해서 서비스.
  임베디드 위젯용 vanilla JS SDK가 뭔지 몰라서 설명이 필요하다.
