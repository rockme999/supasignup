## Round 2 답변에 대한 응답

### "앱스토어를 통해서 어떻게 로그인화면을 바꿀 수 있지?"에 대한 답변

카페24 앱스토어 앱이 회원가입 화면을 수정하는 방식은 2가지입니다:

**방식 1: 스크립트 태그 자동 삽입 (ScriptTag API)**
- 카페24 앱이 설치되면 ScriptTag API로 쇼핑몰 전체에 JS를 자동 삽입할 수 있음
- 운영자가 직접 코드를 붙여넣을 필요 없음
- 경쟁사 "1초가입"도 이 방식을 사용할 가능성이 높음
- 참고: https://developers.cafe24.com/docs/ko/api/admin/#scripttags

**방식 2: 쇼핑몰 디자인 편집에서 수동 삽입**
- 운영자가 직접 카페24 디자인 편집 > 회원가입 페이지에서 코드를 붙여넣는 방식

**결론**: 카페24 앱스토어 등록 시 → ScriptTag API로 자동 삽입 (운영자 작업 0)
독립 배포 시 → 운영자가 직접 코드 붙여넣기

-->  좋아 이 방식은 아주 유용하겠는데 다른 플랫폼의 경우는 어떻게 처리 할지 그 때가서 고민해야 하겠지? 우선은 카페24만 우선 구현 및 서비스 개시를 목표로 하자.

### Q1. 아키텍처 방향 - VeriPack-Multi 구조 재활용

VeriPack-Multi는 플랫폼별 독립 서버(server-cafe24, server-imweb 등) + 공통 core 패키지 구조입니다.
SupaSignup도 동일 구조로 갈까요?

```
supasignup/
├── packages/
│   └── signup-core/          # 공통: OAuth 처리, 회원 생성 로직
├── server-cafe24/            # 카페24 연동
├── server-imweb/             # (추후)
├── server-godomall/          # (추후)
├── server-shopby/            # (추후)
├── widget/                   # 쇼핑몰에 삽입되는 JS (가입 버튼)
└── docs/
```

장점: VeriPack-Multi 코드 재활용 극대화, 익숙한 구조
단점: 서비스 규모 대비 과한 구조일 수 있음

아니면 하나의 서버에서 플랫폼을 분기 처리하는 모놀리식도 가능합니다.

### Q2. 회원 생성 플로우 (핵심 기술 질문)

조사 결과, 각 플랫폼의 **회원 생성(쓰기) API**는 아직 구현/검증되지 않았습니다.
이 부분이 SupaSignup의 성패를 결정하는 핵심입니다.

가능한 시나리오:
- **(A)** 플랫폼 API로 회원 직접 생성 가능 → 가장 이상적
- **(B)** 플랫폼 API로는 불가, 쇼핑몰 프론트엔드에서 JS로 회원가입 폼을 자동 제출 → 가능하지만 불안정
- **(C)** 우리 DB에만 임시 회원 저장, 쇼핑몰에는 주문 시점에 생성 → 가장 안전하지만 "로그인 상태" 구현이 복잡

카페24의 경우 회원 생성 API 존재 여부를 먼저 확인해야 합니다.
제가 카페24 개발자 문서에서 회원 관련 API를 조사해볼까요?

--> 우선은 카페24 부터 서비스 개발하고 다른 플랫폼에 포팅할거니까 우리가 필요로 하는것들을 조사해봐. 그 전에 위에서 언급한데로 cafe24-common라이브러리 부터 살펴봐.
