## 0. 전제 — 모바일 우선 설계

### 왜 모바일 우선인가

| 지표 | 수치 |
|------|------|
| 한국 이커머스 모바일 구매 비율 (2025) | **73.2%** |
| 2025년 전망 | **82%** |
| 모바일 쇼핑 거래액 | **182조원+** |

### 한국 모바일 이커머스의 특수 환경

| 환경 | ScriptTag 영향 |
|------|---------------|
| **카카오톡/네이버 인앱 브라우저** | localStorage 세션 간 미유지 가능, Share API 미지원 |
| **삼성 인터넷 (40대+)** | **광고 차단기 내장** — 클래스명에 ad/popup/banner 사용 시 필터링 위험 |
| **iOS Safari** | ITP 7일 쿠키 제한, Private 모드 localStorage 예외 |
| **Android Chrome** | 가장 안정적 — 기준 구현 대상 |

### ScriptTag 기술 제약

- ScriptTag는 **PC + 모바일 스킨 모두에 삽입됨**
- `display_location`으로 모바일/PC 구분 불가 → JS 런타임에서 분기
- 카페24 앱당 ScriptTag 수 제한 → **단일 loader.js로 모든 기능 관리**

### ScriptTag 필수 기술 스택

```
[반드시 사용]
- Shadow DOM (closed)     → CSS/DOM 격리
- IntersectionObserver    → 뷰포트 감지, 노출 추적
- MutationObserver        → SPA 페이지 전환 감지
- Beacon API              → 이탈 시 분석 데이터 전송
- visibilitychange        → 탭/앱 전환 감지
- matchMedia              → 모바일/PC/다크모드 분기
- visualViewport API      → 키보드/주소창 변화 대응
- localStorage + Cookie   → 이중 저장 (인앱 브라우저 대응)

[사용 금지]
- viewport meta 수정      → 호스트 페이지 레이아웃 파괴
- history.pushState 남용   → 쇼핑몰 라우팅 충돌
- Web Notification 요청    → 사용자 신뢰도 저하
- innerHTML               → XSS 위험
- "ad/banner/popup" 클래스명 → 광고 차단기 필터링
- CSS !important 남발     → 호스트 스타일 파괴
- 전역 변수 오염           → 네임스페이스 격리 필수
```

### 1.3 핵심 발견

1. **ScriptTag로 기존 DOM 자유 조작 가능** — 텍스트, 스타일, 애니메이션, 새 요소 삽입 모두 OK
2. **flex 컨테이너 주의** — `.action_button`이 flex라서 내부 삽입 시 옆으로 붙음 → **부모 레벨에 삽입** 또는 **버튼 내부 absolute**로 해결
3. **모바일 플로팅 버튼 별도 처리 필요** — `#orderFixArea`는 PC 본문 버튼과 별개. 모바일에서는 이 플로팅 버튼이 항상 보이므로 여기를 타겟해야 함
4. **페이지 구분 필수** — 장바구니의 "전체상품주문"과 상품상세의 "BUY IT NOW"이 같은 선택자 (`#orderFixArea a.btnSubmit`). URL/DOM 기반 페이지 판별 필요
5. **오버레이 뱃지 패턴 유효** — 작은 뱃지(알약형)가 버튼 위에 겹치면서 점멸 → 시각적으로 효과적이고 원래 버튼 기능 방해 안 함

### 1.4 카페24 상품 상세 DOM 맵

```
.headingArea h1                          ← 상품명
#span_product_price_custom               ← 소비자가 (취소선)
#span_product_price_text                 ← 판매가
.delv_price_B                            ← 배송비
#product_option_id1                      ← 옵션 선택 (select)
.action_button                           ← 버튼 컨테이너 (flex) — PC 본문
  a.btnSubmit > span#actionBuy           ← BUY IT NOW
  button#actionCart                       ← CART
  button#actionWish                       ← WISH LIST
#orderFixArea                            ← 모바일 하단 플로팅 컨테이너
  a.btnSubmit > span#actionBuy           ← BUY IT NOW (플로팅)
  button.actionCart                       ← CART (플로팅)
```

### 5.4 AI 활동 보고 시스템 — "AI 직원"

> 핵심 개념: AI를 "설치하고 잊는 도구"가 아니라, **매일 보고하는 마케팅 직원**으로 포지셔닝

#### 보고 체계

| 주기 | 보고 내용 | 대상 | 가치 |
|------|----------|------|------|
| **일일 보고** | AI가 수행한 홍보 활동 일지 | 관리자(쇼핑몰 운영자) | "오늘 뭘 했는지" 투명성 |
| **주간 보고** | 주간 성과 요약 (가입 수, 전환율, 클릭률) | 관리자 | "실제로 효과가 있는지" 증명 |
| **월간 보고** | 인사이트 + 경쟁업체 분석 + 개선 제안 | 관리자/경영진 | "다음에 뭘 해야 하는지" 전략 |

#### 일일 보고 예시

```
📋 번개가입 AI 일일 활동 보고 — 2026-04-15 (화)

[오늘의 활동]
• 상품상세 페이지: "3,000원 할인" 뱃지 노출 1,247회, 클릭 89회 (7.1%)
• 장바구니 페이지: 체류 10초+ 고객 42명에게 추가 쿠폰 제공 → 18명 주문 완료
• 미니배너: AI 생성 카피 "봄맞이 가입 혜택" 노출 → 기존 카피 대비 +23% 클릭률
• 소셜 가입: 카카오 31명, 네이버 12명, 구글 8명, 기타 4명

[오늘의 특이사항]
• 카카오 인앱 브라우저 유입 비율 41% (전주 대비 +5%)
• 모바일 전환율이 PC 대비 1.8배 높음
```

#### 주간 보고 예시

```
📊 번개가입 AI 주간 성과 보고 — 2026-04-09 ~ 04-15

[핵심 지표]
• 신규 가입: 385명 (전주 대비 +12%)
• 가입 전환율: 4.2% → 4.8% (+0.6%p)
• 가장 효과적인 효과: 가격 카운트다운 (전환율 6.1%)
• 가장 효과적인 프로바이더: 카카오 (전체의 57%)

[AI 최적화 결과]
• A/B 테스트: 배너 카피 B안 ("회원만 할인") 승리 → 자동 전환 완료
• 버튼 순서: 카카오→네이버→구글 순서가 최적 (자동 적용 중)
```

#### 월간 보고 예시

```
📈 번개가입 AI 월간 인사이트 — 2026년 4월

[성과 요약]
• 누적 가입: 1,580명 / 전환율: 4.8% / 월 매출 기여: 추정 ₩4,740,000

[인사이트]
• 화~목 오후 2~6시가 가입 전환율 최고 (7.2%) → 이 시간대 배너 강화 권장
• 모바일 사용자의 장바구니 체류 시간: 평균 23초 → 10초 에스컬레이션이 최적
• 카카오 가입자의 구매 전환율(32%)이 네이버(18%)보다 1.8배 높음

[경쟁 분석]
• 1초가입: 이번 달 앱스토어 리뷰 3건 추가 (부정 1건 — UI 오류 관련)
• 알파푸시: 새 기능 "AI 상품 추천" 출시 — 우리와 직접 경쟁 아님
• 시장 동향: 카페24 앱스토어 "소셜 로그인" 카테고리 검색량 +15%

[다음 달 제안]
• Discord 프로모션 강화 (MZ 유입 채널로 성장 중)
• 무료배송 게이지 활성화 권장 (장바구니 이탈 32% → 예상 20%로 감소)
```

#### 이 시스템의 전략적 의의

**1. 신뢰 구축 → 해지 방어**
- 운영자가 "이 앱이 실제로 일하고 있다"고 매일 느낌
- 경쟁사는 설치 후 아무런 피드백 없음 → "효과가 있나?" 의심 → 해지
- 번개가입은 매일 보고 → "이렇게 열심히 하는데 끊을 수 없지"

**2. 데이터 Lock-in 심화**
- 일일/주간/월간 보고가 쌓일수록 → "이 데이터를 버릴 수 없다"
- 3개월치 보고서 = 쇼핑몰의 고객 행동 패턴 자산
- 경쟁사로 이전하면 이 모든 인사이트가 소멸

**3. 경쟁사 불가 차별점**
- "AI가 일일보고하는 마케팅 도구" → 카페24 앱스토어에 전무
- 앱스토어 설명: "매일 보고하는 AI 마케터를 월 0원에 고용하세요"
- 이것 자체가 마케팅 메시지

**4. 상위 플랜 전환 근거**
- 무료 플랜: 기본 일일 요약만
- 유료 플랜: 상세 분석 + 경쟁 분석 + AI 제안
- 보고서 품질 차이 = 자연스러운 업그레이드 동기

#### Phase별 구현

| Phase | 보고 수준 | 구현 |
|-------|----------|------|
| **MVP (Phase 1)** | 기본 일일 요약 (가입 수, 클릭률) | 대시보드 내 간단 리포트 |
| **V2 (Phase 2)** | 주간 성과 보고 + AI 최적화 결과 | 이메일/카카오 알림톡 발송 |
| **V3 (Phase 3)** | 월간 인사이트 + 경쟁 분석 + AI 제안 | 자동 생성 리포트 (AI) |

## 11. 상품 열람 이력 기반 행동 추론 시스템 (2026-04-01 추가)

> **핵심 아이디어**: ScriptTag는 모든 상품 상세 페이지에서 실행된다. 사용자가 어떤 상품을 봤는지, 얼마나 봤는지, 어떤 순서로 봤는지를 **localStorage에 쌓아두면** — 그 패턴으로 "이 사람이 뭘 원하는지"를 꽤 정확하게 추론할 수 있다.

### 11.1 수집 가능한 데이터 (DOM에서 추출)

ScriptTag가 상품 상세 페이지 진입 시 자동 수집하는 데이터:

```javascript
// 상품 상세 페이지에서 DOM 파싱으로 수집 가능한 데이터
const productData = {
  product_no: extractFromURL(),                    // URL: /product/상품명/123/
  name: document.querySelector('.headingArea h1')?.textContent,
  price: parsePrice('#span_product_price_text'),   // 판매가
  original_price: parsePrice('#span_product_price_custom'), // 소비자가 (할인 여부 판별)
  category: extractCategoryFromBreadcrumb(),       // 네비게이션 경로
  brand: document.querySelector('.brand')?.textContent,
  options_viewed: [],                              // 선택한 옵션 추적
  has_discount: original_price > price,            // 할인 상품 여부
  shipping_fee: parsePrice('.delv_price_B'),       // 배송비
  timestamp: Date.now(),
  time_spent: 0,                                   // 이탈 시 갱신
  scroll_depth: 0,                                 // 최대 스크롤 깊이
  actions: []                                      // 장바구니 담기, 옵션 선택 등
};
```

### 11.2 행동 패턴 추론 엔진

수집된 데이터로 다음 패턴을 추론:

#### A. 구매 의도 점수 (Purchase Intent Score)

```
[행동]                              [점수]
상품 상세 진입                        +1
체류 30초 이상                        +2
스크롤 80% 이상 (리뷰까지 봄)          +3
옵션 선택함                           +5
같은 상품 재방문                       +8
장바구니 담기 후 미결제                 +10
---
0~3점: 구경 중 (Window Shopping)
4~8점: 관심 있음 (Interest)
9~15점: 구매 고민 중 (Consideration)
16점+: 구매 직전 (Ready to Buy)
```

#### B. 가격 민감도 추론

| 행동 패턴 | 추론 | 대응 전략 |
|----------|------|----------|
| 같은 카테고리 3개+ 상품을 **가격 높→낮** 순서로 봄 | 예산 맞추는 중 | "회원가로 보면 이 상품도 예산 내!" |
| **할인 상품만** 연속 클릭 | 가격 민감 고객 | 할인 쿠폰 강조, 가격 카운트다운 최대 강도 |
| 고가 상품 1개를 **3회 이상 재방문** | 비싸서 고민 중 | "가입하면 N원 할인 + 무이자 할부 안내" |
| 가격대 분산 없이 **일정 범위**만 봄 | 명확한 예산 보유 | 해당 가격대 인기 상품 추천 |
| 소비자가(취소선) 있는 할인 상품에만 반응 | 할인 심리 강함 | "회원 전용 추가 할인" 배너 |

#### C. 카테고리 관심도 분석

```
수집 데이터:
  상의/셔츠: 3회 열람, 평균 체류 35초
  하의/팬츠: 2회 열람, 평균 체류 20초
  아우터:    1회 열람, 체류 8초

추론:
  → 주 관심: 셔츠 (빈도 + 체류시간 모두 최고)
  → 부 관심: 팬츠 (빈도 있음)
  → 스쳐지나감: 아우터 (빈도 낮고 체류 짧음)
```

#### D. 쇼핑 스타일 분류

| 스타일 | 판별 조건 | 특징 |
|--------|----------|------|
| **탐색형** (Browser) | 5개+ 상품, 각 체류 15초 미만, 옵션 선택 없음 | 빠르게 훑어보는 중 |
| **비교형** (Comparator) | 같은 카테고리 3개+, 체류 30초+, 스크롤 깊음 | 꼼꼼히 비교 중 |
| **목적형** (Decider) | 1~2개 상품만, 체류 1분+, 옵션 선택함 | 이미 뭘 살지 알고 있음 |
| **재방문형** (Returner) | 같은 상품 2회+ 방문, 간격 1시간+ | 고민하다 돌아옴 |
| **충동형** (Impulser) | 다양한 카테고리, 할인 상품 집중, 빠른 장바구니 | 세일/할인에 반응 |

### 11.4 기술 구현 — 데이터 수집기 핵심 코드 구조

```javascript
// modules/browsing-tracker.js

class BrowsingTracker {
  constructor() {
    this.STORAGE_KEY = 'bg_browsing_history';
    this.MAX_ITEMS = 50;
    this.sessionStart = Date.now();
    this.scrollPositions = [];
    this.lastInteraction = null;
  }

  // 상품 상세 페이지 진입 시 자동 호출
  trackProductView() {
    const product = this.extractProductFromDOM();
    if (!product) return;

    const history = this.getHistory();
    const existing = history.find(p => p.id === product.id);

    if (existing) {
      existing.visit_count++;
      existing.last_visited = Date.now();
      existing.price_history.push({ price: product.price, at: Date.now() });
    } else {
      history.unshift({
        ...product,
        first_visited: Date.now(),
        last_visited: Date.now(),
        visit_count: 1,
        time_spent: 0,
        max_scroll: 0,
        actions: [],
        price_history: [{ price: product.price, at: Date.now() }]
      });
    }

    // FIFO: 50개 초과 시 오래된 것 삭제
    if (history.length > this.MAX_ITEMS) history.pop();
    this.saveHistory(history);

    // 이탈 시 체류시간 + 스크롤 깊이 업데이트
    this.startEngagementTracking(product.id);
  }

  // DOM에서 상품 데이터 추출
  extractProductFromDOM() {
    const path = window.location.pathname;
    const productMatch = path.match(/\/product\/[^/]+\/(\d+)/);
    if (!productMatch) return null;

    return {
      id: productMatch[1],
      name: this.getText('.headingArea h1'),
      price: this.parsePrice('#span_product_price_text'),
      original_price: this.parsePrice('#span_product_price_custom'),
      category: this.extractCategory(),
      shipping: this.getText('.delv_price_B'),
      url: window.location.href,
      image: document.querySelector('.thumbnail img')?.src
    };
  }

  // 행동 패턴 분석
  analyzePatterns() {
    const history = this.getHistory();
    if (history.length < 2) return null;

    return {
      shopping_style: this.classifyStyle(history),
      price_sensitivity: this.analyzePriceSensitivity(history),
      category_interest: this.analyzeCategoryInterest(history),
      purchase_intent: this.calculatePurchaseIntent(history),
      time_pattern: this.analyzeTimePattern(history),
      cross_sell: this.detectCrossSellOpportunity(history),
      exit_context: this.getExitContext()
    };
  }

  // 쇼핑 스타일 분류
  classifyStyle(history) {
    const recentSession = history.filter(p =>
      Date.now() - p.last_visited < 30 * 60 * 1000 // 30분 내
    );
    const avgTimeSpent = avg(recentSession.map(p => p.time_spent));
    const hasRevisits = recentSession.some(p => p.visit_count > 1);
    const categoryCount = new Set(recentSession.map(p => p.category)).size;

    if (recentSession.length > 5 && avgTimeSpent < 15) return 'browser';
    if (hasRevisits && avgTimeSpent > 30) return 'comparator';
    if (recentSession.length <= 2 && avgTimeSpent > 60) return 'decider';
    if (hasRevisits && recentSession.some(p =>
      Date.now() - p.first_visited > 60 * 60 * 1000
    )) return 'returner';
    return 'impulser';
  }
}
```

### 11.5 프라이버시 & 윤리적 고려

| 항목 | 방침 |
|------|------|
| **데이터 저장 위치** | 100% localStorage (사용자 기기에만 존재) — 서버 전송 안 함 |
| **서버 전송 조건** | 사용자가 가입한 후 명시적 동의 시에만 서버 동기화 |
| **데이터 보존 기간** | localStorage 최대 50개, 30일 이상 된 항목 자동 삭제 |
| **추론 결과 활용** | 메시지 개인화에만 사용, 외부 광고 플랫폼에 제공 안 함 |
| **투명성** | "이 추천은 귀하의 열람 이력을 기반으로 합니다" 명시 |
| **옵트아웃** | 설정에서 "개인화 추천 끄기" 가능 |
| **가짜 데이터 방지** | 재고/판매 카운트는 실제 데이터 또는 명확히 "참고용" 표시 |

## 12. 공통 인프라 명세 — 행동 추론 시스템의 기반 (2026-04-01 추가)

> **왜 이게 먼저인가**: R1~R10 어떤 기능을 구현하든 아래 3가지가 선행되어야 한다.
> 1. **사용자 행동 수집/분류 체계** — 뭘 기록하고 어떻게 분류할 것인가
> 2. **쇼핑몰 초기 설정(온보딩)** — 쇼핑몰 특성/업종/혜택 범위를 파악

### 12.1 쇼핑몰 초기 설정 (온보딩)

앱 설치 직후, 쇼핑몰 관리자에게 최소한의 정보를 입력받아야 AI가 맥락에 맞는 행동을 할 수 있다.

#### A. 자동 수집 (API scope 기반 — 관리자 입력 불필요)

### 12.3 관리자 혜택 설정 체계

> **핵심 원칙**: 관리자는 "예산과 규칙"을 정하고, AI는 그 안에서 최적의 타이밍/대상/조합을 결정한다.

#### A. 혜택 종류 정의

```
[온보딩 Step 4: 혜택 설정]

가입 시 제공할 혜택을 설정하세요 (복수 선택 가능):

□ 적립금/마일리지
  └ 금액: [______] 원  (예: 3,000)

□ 할인 쿠폰
  └ 할인 유형: ○ 정액 [______]원  ○ 정률 [___]%
  └ 최소 주문금액: [______] 원
  └ 유효기간: [___] 일

□ 무료배송 쿠폰
  └ 최소 주문금액: [______] 원  (0이면 무조건)
  └ 유효기간: [___] 일

□ 추가 혜택 (AI 재량)
  └ AI가 사용할 수 있는 월간 예산: [______] 원
  └ 건당 최대: [______] 원
  └ 용도: □ 재방문 유도  □ 이탈 방지  □ 첫 구매 촉진  □ 전체
```

#### B. 혜택 풀 (Benefit Pool) 구조

```typescript
// 관리자가 설정한 혜택 풀 — API로 관리
interface BenefitPool {
  shop_id: string;

  // 가입 기본 혜택 (항상 제공)
  signup_benefits: {
    mileage?: { amount: number };                           // 적립금
    coupon?: {
      type: 'fixed' | 'percent';
      value: number;
      min_order: number;
      valid_days: number;
    };
    free_shipping?: { min_order: number; valid_days: number };
  };

  // AI 재량 혜택 (상황에 따라 AI가 선택적 제공)
  ai_discretionary: {
    enabled: boolean;
    monthly_budget: number;        // 월간 예산 (원)
    per_action_max: number;        // 건당 최대 (원)
    remaining_budget: number;      // 이번 달 잔여 예산
    allowed_purposes: Purpose[];   // 허용된 용도
  };

  // 혜택 에스컬레이션 규칙 (AI 자율도에 따라)
  escalation: {
    mode: 'conservative' | 'balanced' | 'aggressive';
    // conservative: 고정 혜택만, 분기 없음
    // balanced: 기본 혜택 + 상황별 메시지 차별화 (혜택 금액은 고정)
    // aggressive: AI가 예산 내에서 혜택 금액/종류 동적 조절
  };
}

type Purpose = 'revisit' | 'exit_prevention' | 'first_purchase' | 'all';
```

#### C. AI 재량 혜택의 사용 시나리오

```

  가입 적립금: 3,000원
  AI 월 예산: 50,000원 (건당 최대 1,000원)
  용도: 이탈 방지 + 재방문 유도

AI 행동:
  ① 일반 가입 유도  → "가입하면 3,000원 적립금!" (기본 혜택만 안내)
  ② 이탈 감지      → "지금 가입하면 3,000원 + 추가 1,000원!" (AI 예산 사용)
  ③ 3회+ 재방문     → "돌아오셨네요! 오늘만 500원 추가 적립" (AI 예산 사용)
  ④ 예산 소진 시    → 기본 혜택만 안내 (추가 혜택 중단)

AI 보고:
  "이번 달 AI 재량 예산 50,000원 중 32,000원 사용
   - 이탈 방지: 18건, 12,000원 → 가입 전환 7건 (38.9%)
   - 재방문 유도: 40건, 20,000원 → 가입 전환 15건 (37.5%)
   - 투자 대비 효과: 32,000원 투자 → 22건 추가 가입"
```

### 12.4 R1 판단 — "옵션화 vs 분기 로직"

> 결론: **하이브리드** — 관리자는 "허용 범위"를 토글, AI는 그 안에서 자동 분기

#### 세 가지 방식 비교

| | 옵션화 (토글) | 분기 로직 (자동) | **하이브리드 (권장)** |
|---|---|---|---|
| **관리자 역할** | 개별 기능 ON/OFF | 없음 (방치) | 범위 설정 + 결과 확인 |
| **AI 역할** | 없음 (기계적 실행) | 전부 자동 | 범위 내 자율 판단 |
| **구현 복잡도** | 하 | 상 | 중 |
| **최적화 가능** | ✗ (고정) | ✅ (무제한) | ✅ (범위 내) |
| **관리자 신뢰도** | 높음 (직접 통제) | 낮음 (블랙박스) | 높음 (보고로 투명) |
| **AI 직원 포지셔닝** | ✗ (로봇) | △ (무책임) | **✅ (보고하는 직원)** |

### 13.1 R3 개선 — 비회원 최근 본 상품 위젯

> **질문**: 로컬 저장 vs 서버 저장?
> **답변**: **2단계 전략** — 가입 전 로컬, 가입 후 서버 동기화

```
[Phase 1: 비회원 — 100% 로컬]

  ScriptTag → 상품 열람 시 localStorage에 자동 저장
  위젯에서 localStorage 데이터를 직접 읽어서 표시
  → 서버 통신 0, 비용 0, 프라이버시 이슈 0

[Phase 2: 가입 순간 — 로컬 → 서버 동기화]

  가입 완료 이벤트 감지
  → localStorage의 열람 이력을 서버에 일괄 전송
  → 서버 DB에 member_id와 연결하여 저장
  → "기기 바꿔도 이어서 쇼핑" 가능

[Phase 3: 회원 — 서버 기반 + 로컬 캐시]

  서버에 열람 이력 저장 (API로 전송)
  로컬에도 캐시 유지 (오프라인/속도)
  다른 기기에서 로그인 → 서버에서 이력 불러옴

핵심 가치:
  비회원: "가입하면 이 목록을 잃지 않습니다" (손실 회피)
  회원: "어디서든 이어서 쇼핑" (편의성)
  관리자: "비회원의 관심 상품 데이터" (가입 전환 시 마케팅 활용)
```

### 13.2 R4 개선 — 이탈 맥락 감지

> **질문 1**: 스크롤 추적 가능한가?
> **답변**: **가능** — IntersectionObserver + scroll 이벤트

```javascript
// 스크롤 존 감지 — 상품 상세 페이지 4구간
// ScriptTag에서 완전히 구현 가능

// 방법 1: IntersectionObserver (권장 — 성능 최적)
const zones = {
  top:    document.querySelector('.headingArea'),      // 상품 이미지
  info:   document.querySelector('#span_product_price_text'), // 가격/옵션
  detail: document.querySelector('.detail'),           // 상세 설명
  review: document.querySelector('.review')            // 리뷰
};

Object.entries(zones).forEach(([zone, el]) => {
  if (!el) return;
  new IntersectionObserver((entries) => {
    if (entries[0].isIntersecting) {
      currentScrollZone = zone;  // 현재 사용자가 보고 있는 영역
    }
  }, { threshold: 0.5 }).observe(el);
});

// 방법 2: scroll 이벤트 (fallback)
window.addEventListener('scroll', throttle(() => {
  const depth = window.scrollY / document.body.scrollHeight;
  if (depth < 0.25) currentScrollZone = 'top';
  else if (depth < 0.50) currentScrollZone = 'info';
  else if (depth < 0.75) currentScrollZone = 'detail';
  else currentScrollZone = 'review';
}, 200));
```

> **질문 2**: 옵션 선택 후 장바구니 자동 담기의 의미?
> **답변**: Front API로 **기술적으로 가능**. 단, 행동의 의미를 재정의

```
[기존 안 — 문제점]
  옵션 선택 → 이탈 → "가입하면 장바구니에 담아줄게요"
  → 문제: 너무 공격적, "내가 선택한 게 어떻게 저장되지?" 불안감

[개선안 — 2단계 접근]

  Step 1 (가입 전):
    옵션 선택 감지 → localStorage에 기록
    이탈 시 메시지:
    "L사이즈/블랙 고르셨죠? 가입하면 다음에 다시 안 골라도 돼요"
    → 기능: "기억해줄게" (부드러운 유도)

  Step 2 (가입 후):
    가입 완료 → "선택하셨던 L/블랙을 장바구니에 담아드릴까요?"
    → [담기] [괜찮아요]  ← 사용자 동의 후 Front API 장바구니 추가
    → 동의 기반이므로 불안감 없음
```

> **핵심**: 이탈 맥락 감지의 **합리적 판단 기준**

```
[맥락 감지 판단 매트릭스]

  이탈 시점의 상태          → 추론              → 메시지 방향       → 신뢰도
  ─────────────────────────────────────────────────────────────────────────
  스크롤 top (이미지만 봄)  → 관심 없음         → 가볍게 혜택만 언급 → 중
  + 체류 5초 미만

  스크롤 info (가격 확인)   → 가격 부담         → 할인/혜택 강조    → 높음
  + 체류 10초+

  스크롤 info + 옵션 선택   → 구매 직전 이탈    → 옵션 기억 + 혜택  → 매우 높음
  + 장바구니 미담기

  스크롤 detail (상세 봄)   → 상품 비교 중      → "비교 저장" 제안  → 중
  + 체류 30초+

  스크롤 review (리뷰 봄)   → 신뢰 부족         → 사회 증명 강조    → 높음
  + 체류 20초+

  배송비 영역 확인 후 이탈  → 배송비 부담       → 무료배송 쿠폰     → 매우 높음

  여러 상품 빠르게 훑기     → 원하는 것 못 찾음 → 추천/검색 안내    → 중
  + 각 체류 10초 미만
  ─────────────────────────────────────────────────────────────────────────

  [신뢰도별 대응]
  매우 높음 → 즉시 맥락 메시지 표시
  높음     → 맥락 메시지 표시 (약간 부드럽게)
  중       → 일반 가입 유도 (맥락 추론 안 함 — 틀릴 위험)
  낮음     → 메시지 표시 안 함 (노이즈 방지)
```

### 13.4 R6 개선 — 시간 패턴 + 배송/재고 정보

> **질문**: 배송 주기/재고 파악에 관리자 입력 vs 상품 scope?
> **답변**: **둘 다** — 기본은 API 자동, 세부는 관리자 보정

```
[자동 (API 기반)]
  mall.read_product → 상품별 재고 수량 실시간 조회 가능
  → "재고 3개 남음"은 실제 데이터 기반으로 표시 가능
  → 가짜 재고 카운트다운 불필요 — 진짜 데이터가 더 강력

[관리자 입력 (온보딩 Step 3)]
  배송 소요일, 무료배송 기준, 당일배송 여부
  → "지금 주문하면 내일 도착!" 정확도 보장

[하이브리드]
  API 재고 < 5 → 자동으로 "재고 N개 남음" 표시 (관리자 별도 설정 불필요)
  API 재고 > 5 → 재고 메시지 표시 안 함 (거짓 긴급성 방지)
  배송일 = 관리자 입력값 + 주문 시각 기반 계산

  R6 시간 패턴과 결합:
    심야 주문 → "밤에 주문해도 D+{관리자 입력}일 도착"
    오전 주문 → "지금 주문하면 D+{관리자 입력-1}일 도착!" (당일 발송 시)
```

### 13.5 R7 개선 — 구매 여정 게이미피케이션

> **피드백**: 너무 자주 노출 시 신선도 하락, 결정적 시점에만 표시, AI 보고 자료로 활용

```
[노출 규칙 — "결정적 시점"만]

  노출 조건 (AND):
    ① 구매 의도 점수 >= 9 (Consideration 이상)
    ② 현재 세션에서 아직 표시되지 않음 (세션당 최대 1회)
    ③ 직전 행동이 "진전"에 해당 (옵션 선택, 장바구니 추가 등)

  노출하지 않는 경우:
    - 탐색형(browser) 고객 → 아직 관심 단계 미도달
    - 이미 장바구니에 상품 있는 고객 → 여정 표시 불필요
    - 이전 세션에서 이미 본 고객 → 72시간 쿨다운

  목적별 분기:
```

```
    - "매출의 2%만 쓴다"는 심리적 안정감

  구현:
    - Admin API 주문 데이터로 전월 매출 자동 집계
    - 월초에 예산 자동 계산 → 관리자에게 알림
    - "이번 달 AI 예산: 120,000원 (전월 매출 600만원의 2%)"
```

### 16.5 목표 달성 시나리오별 AI 행동

```
[시나리오 1: 목표 초과 달성 중 — 예산 절약 모드]

  목표: 월 50명 가입 / 예산: 50,000원
  중간 점검 (15일차): 이미 42명 가입, 예산 28,000원 사용

  AI 판단:
  "목표 달성 확실 → 나머지 기간 예산 절약 모드
   → 추가 혜택 제공 줄이고 메시지 최적화에 집중
   → 절약된 예산은 다음 달로 이월 또는 보조 목표에 투입"

[시나리오 2: 목표 미달 위험 — 집중 투자 모드]

  목표: 월 50명 가입 / 예산: 50,000원
  중간 점검 (15일차): 18명 가입, 예산 20,000원 사용

  AI 판단:
  "목표 달성 어려움 → 남은 30,000원을 고효율 채널에 집중
   → 이탈 방지에 70% 집중 (가장 높은 전환율)
   → 관리자에게 알림: '예산 추가 투입 시 목표 달성 가능성 ↑'"

[시나리오 3: 예산 소진 — 무비용 모드]

  예산 전액 소진 (25일차)

  AI 행동:
  "추가 혜택 제공 중단 → 텍스트 메시지만으로 가입 유도
   → 기존 기본 혜택(가입 적립금)은 계속 안내
   → AI 재량 추가 혜택만 중단
   → 보고: '예산 소진. 무비용 메시지로 전환. 추가 예산 권장?'"
```

```javascript
  'mall.read_order',        // 주문 이력

  'mall.read_promotion',    // 쿠폰/혜택 조회
  'mall.write_promotion',   // 쿠폰 발급
  'mall.read_shipping',     // 배송 설정
].join(','));

// ※ 적립금 지급 시 scope 부족(403) 발생하면 mall.write_mileage 추가
```

### 17.5 웹훅 + ScriptTag 역할 분담

```
[이벤트 감지 — 어디서 하는가?]

                        ScriptTag (프론트)    Webhook (서버)
                        ──────────────────    ──────────────
페이지 방문/이탈             ✅                    ✗
스크롤/체류시간              ✅                    ✗
옵션 선택/장바구니 클릭      ✅                    ✗
이탈 의도 감지               ✅                    ✗
배너/뱃지 노출/클릭          ✅                    ✗

회원 가입 완료               △ (불확실)            ✅ (확실)
주문 생성/결제               ✗                    ✅
주문 취소/환불               ✗                    ✅
상품 가격 변경               ✗                    ✅
상품 품절                    ✗                    ✅
앱 삭제                      ✗                    ✅
회원 탈퇴                    ✗                    ✅

→ ScriptTag = 사용자 행동 (실시간, 프론트)
```

### 17.7 scope 추가 시 주의사항
