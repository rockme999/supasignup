## 1. 전역 자동답변 토글 켜고 끄기

### 위치

`/supadmin/inquiries` → 상단 토글 영역

### 켜는 절차

1. **사전 확인**: 수동 모드로 최소 1~2주 운영해 답변 품질을 충분히 검증한 뒤 전환
2. `/supadmin/inquiries` 접속
3. "전역 AI 자동답변" 토글을 **ON** 으로 전환
4. 경고 메시지 확인 후 확인 클릭

> **⚠️ 주의**: ON 상태에서 새 문의가 들어오면 AI가 즉시(30초~1분 이내) 고객에게 자동 발송됩니다.  
> 답변이 잘못되어도 이미 고객이 받은 상태이므로, 켜기 전에 품질 검증이 선행되어야 합니다.

### 끄는 절차

1. `/supadmin/inquiries` 접속
2. 토글을 **OFF** 로 전환
3. 변경 즉시 반영 (이미 진행 중인 자동답변은 완료됨)

### 확인 방법

- 토글 UI에서 현재 상태를 즉시 확인 가능
- `PUT /api/supadmin/settings/auto-reply` 변경 시 감사 로그 자동 기록됨  
  → `/supadmin/audit-logs` 에서 `toggle_global_auto_reply` 액션으로 이력 조회 가능

## 2. 뱃지 의미 해석

문의 목록 각 행의 상태 뱃지 오른쪽에 추가로 표시됩니다.

| 뱃지 | 색상 | 의미 | 해야 할 일 |
|---|---|---|---|
| 🔴 **AI 답변 미검수** | 빨강 | `auto_replied` 상태인데 관리자가 아직 내용을 열어보지 않음 | **가장 우선 확인**. 모달 열어서 AI 답변 내용 검수 |
| 🟠 **미확인** | 노랑 | 수동 답변(`replied`)인데 관리자가 아직 확인 안 함 | 내용 확인 후 필요 시 수정 |
| 👁 **쇼핑몰 미열람** | 회색 | 답변이 있는데 쇼핑몰 운영자(고객)가 아직 답변 페이지를 방문하지 않음 | 여유 있을 때 확인. 고객에게 답변 알림 발송 고려 |

**뱃지 우선순위**: 🔴 > 🟠 > 👁 (동시에 여러 조건에 해당하면 가장 높은 우선순위만 표시)

### 읽음 처리 시점

- 🔴, 🟠 뱃지: 관리자가 해당 문의 상세 모달을 **열면** 자동으로 `admin_read_at` 기록 → 뱃지 사라짐
- 👁 뱃지: 쇼핑몰 운영자가 답변 상세 페이지를 **방문하면** 자동으로 `customer_read_at` 기록 → 뱃지 사라짐

## 3. 실패 이력 섹션 읽는 법

문의 상세 모달 하단 "AI 자동답변 실패 이력" 섹션 (실패가 있는 문의에만 표시).

### reason 코드 해석

| reason | 의미 | 조치 |
|---|---|---|
| `ai_error` | Kimi 모델 호출 자체 실패 (일시적 불가) | 1회 자동 재시도. 재시도도 실패하면 수동 답변 |
| `validation_failed` | AI 응답에 금지 토큰·길이 위반 포함 | 1회 자동 재시도. 계속 실패하면 프롬프트 점검 필요 (개발팀 보고) |
| `held_for_review` | AI가 "확인 후 안내" 등 리스크 문구 포함 → 의도된 보류 | 정상 동작. 수동으로 답변 작성 또는 "AI 초안 생성" 버튼 사용 |
| `inquiry_not_found` | 문의 조회 시 이미 `pending`이 아님 (이미 처리됨) | 무시. 다른 경로로 이미 처리됨 |
| `shop_not_found` | 쇼핑몰 정보 조회 실패 | 개발팀 확인 필요 |
| `unexpected_error` | 예상치 못한 오류 | 개발팀 즉시 보고 |

### 실패 이력 예시

```
attempt=1, reason=ai_error, detail="Workers AI timeout", created_at=2026-04-22 09:01:00
attempt=2, reason=held_for_review, detail="keyword_match", created_at=2026-04-22 09:01:04
```

→ 해석: 1차 호출 실패(타임아웃), 3초 후 재시도 → AI가 "확인 후 안내" 포함 → 수동 처리 필요

## 4. AI 답변 미검수 문의 검수 플로우

`🔴 AI 답변 미검수` 뱃지가 있는 문의를 처리하는 표준 절차:

```
1. 문의 목록에서 🔴 뱃지 클릭 → 상세 모달 열기
   (열면 admin_read_at 기록 → 뱃지 자동 제거)

2. 기존 답변 영역에서 AI가 생성한 내용 확인
   → 보라색 박스: "🤖 AI 자동 답변입니다" 디스클레이머 확인

3. 내용 검수:
   ✅ 내용이 정확하고 문제 없음 → 그대로 유지 (고객은 이미 받음)
   ⚠️ 내용이 부정확하거나 보완 필요 → "답변 수정" 으로 덮어쓰기
   ❌ 잘못된 정보 포함 → 즉시 수정 후 고객에게 별도 정정 안내 고려

4. 심각한 오답이 반복되면 → 전역 토글 OFF + 개발팀 프롬프트 점검 요청
```

> **AI 디스클레이머**: 고객이 보는 답변 페이지에도 "이 답변은 AI가 자동으로 생성한 내용입니다" 안내가 표시됩니다. 운영팀 검토 전임을 고객도 인지하고 있습니다.

## 5. 수동 초안 생성 → 검토 → 발송 팁

전역 자동답변이 **OFF** 상태이거나, 자동답변이 보류(`held_for_review`)된 문의에 사용.

### 절차

1. 문의 상세 모달 → 답변 영역 위쪽 **"⚡ AI 답변 초안 생성"** 버튼 클릭
2. 30~60초 기다림 (AI 처리 시간)
3. textarea에 초안이 자동 입력됨

### 검토 체크리스트

- [ ] 사실 관계가 맞는가? (현재 서비스 기능 기준)
- [ ] 고객의 질문에 직접 답변하고 있는가?
- [ ] 불확실한 내용을 단정적으로 말하고 있지 않은가?
- [ ] 마무리 서명 "감사합니다. / 번개가입 드림 ⚡" 포함되어 있는가?
- [ ] 내부 정보(경로, ID, 스키마 등)가 노출되지 않는가?

### 팁

- AI 초안을 그대로 쓰기보다 **고객의 특수한 상황**을 반영해 한두 문장 추가하면 만족도가 높아짐
- 미지원 기능 문의는 AI가 "(1) 현재 미지원 → (2) 대안 → (3) 로드맵 검토 중" 구조로 잘 답변함
- "확인 후 안내드리겠습니다" 유형이면 AI가 held_for_review로 보류함 → 실제 확인 후 직접 작성

## 6. 문제 상황별 조치

### 자동답변이 작동하지 않음

1. `/supadmin/inquiries` 상단 토글이 **ON**인지 확인
2. 해당 문의의 실패 이력 섹션 확인 (모달 하단)
3. `reason=ai_error`가 반복되면 Cloudflare 상태 확인: https://www.cloudflarestatus.com
4. 지속되면 전역 토글 OFF → 개발팀 보고

### 고객이 "답변을 받지 못했다"고 함

1. 문의 상세 확인 → `status` 확인
   - `auto_replied` 또는 `replied` → 답변 발송됨
   - `pending` → 미발송 (자동답변 실패 또는 토글 OFF 상태)
2. `👁 쇼핑몰 미열람` 뱃지 여부 확인 → 뱃지가 있으면 고객이 답변 페이지를 방문하지 않은 것
3. 카카오 채널 등 외부 채널로 "문의 답변이 등록되었습니다" 알림 발송 고려

### AI 응답이 이상함 (잘못된 정보)

1. 즉시 전역 토글 **OFF**
2. 해당 문의 답변 수동 수정
3. 어떤 질문에서 어떤 답변이 나왔는지 캡처해 개발팀 보고
4. 개발팀이 프롬프트/KB 수정 후 재검증 → 통과 후 다시 ON

### AI 응답이 너무 느림 (1분 이상)

- 정상: 캐시 미스(첫 호출) 시 최대 90초 소요될 수 있음
- `auto_replied` 상태가 되는 시간이 고객 문의 제출 후 1~2분 이내면 정상
- 그 이상이면 Cloudflare Workers AI 서비스 지연 가능성 → 상태 페이지 확인

## 7. 월 비용 모니터링

### 확인 경로

```
Cloudflare 대시보드 → Workers & Pages → AI → Analytics
```

### 확인 지표

| 지표 | 정상 범위 (월 600건 기준) | 주의 기준 |
|---|---|---|
| 총 호출 수 | ~600회/월 | 예상보다 2배 이상 → 조사 필요 |
| Neurons 사용량 | ~10.5M/월 | 20M 초과 → 비용 급증 주의 |
| 월 청구액 | $10~15 | $30 초과 → 개발팀 보고 |

### 모니터링 주기

- **주간**: AI > Analytics에서 일별 호출 수 추이 확인
- **월말**: Billing > Invoice 에서 실 청구액 교차 확인
  - 대시보드 neurons 추산과 실 청구서가 다를 수 있음
  - **실 청구서가 기준**

### 비용 급증 원인 & 대응

| 원인 | 대응 |
|---|---|
| 자동답변 켜진 상태에서 문의량 급증 | 토글 OFF 후 원인 파악 |
| 수동 초안 생성 반복 호출 (개발/테스트) | 일일 호출 제한(KV) 확인 |
| 호출 오류 후 재시도 반복 | 실패 이력 확인, 모델 장애 여부 점검 |

## 8. 프롬프트 버전 관리

### 현재 버전

- **프롬프트 버전**: `v1.2-2026-04-22`
- **AI 모델**: `@cf/moonshotai/kimi-k2.5`
- **버전 확인 경로**: `workers/api/src/routes/ai.ts` → `PROMPT_VERSION` 상수

### 버전이 올라가는 경우

1. KB(`kb-public.md`) 내용 추가·수정
2. 답변 규칙(`answer_rules`) 변경
3. 금지 정보 목록(`banned_info`) 업데이트
4. 가드레일 키워드 변경

### 버전 업 절차 (개발팀)

```
1. kb-public.md 또는 프롬프트 내용 수정
2. build-ai-context.mjs 실행 → src/ai-context/*.ts 번들 재생성
3. ai.ts의 PROMPT_VERSION 상수 증가 (예: v1.2 → v1.3)
4. 벤치마크 재실행: node scripts/benchmark-kimi-cache.mjs
5. 품질 평가 (4축 루브릭) → Go 기준 충족 확인
6. 프로덕션 배포
```

### KB 업데이트가 필요한 시점

- 새 소셜 프로바이더 출시 (LINE, X, Facebook 등)
- 요금제 변경
- 고객 문의에서 반복적으로 등장하는 신규 패턴 발견
- 서비스 정책 변경
- 오답이 발견된 경우 (KB 내용이 틀렸거나 누락된 경우)
