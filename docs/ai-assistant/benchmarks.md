# Kimi K2.5 캐시·비용 실측 벤치마크

> 자동 문의 답변용 프롬프트의 실제 Cloudflare Workers AI 동작 기록.
> 새 실측을 돌릴 때마다 `scripts/benchmark-kimi-cache.mjs` 를 실행하면 이 파일 아래에 자동으로 추가됩니다.

## 실행 방법

1. **Cloudflare API Token 발급** (최초 1회)
   - https://dash.cloudflare.com/profile/api-tokens → "Create Token"
   - "Workers AI" > "Read" 권한 템플릿 선택
   - 토큰 복사

2. **환경파일 준비** (최초 1회)
   ```bash
   cp scripts/.env.bench.example scripts/.env.bench
   # scripts/.env.bench 파일을 열고 CF_API_TOKEN= 뒤에 발급받은 토큰 붙여넣기
   ```
   - 이 파일은 `.gitignore` 에 포함됨 → Git에 올라가지 않음

3. **실측 실행**
   ```bash
   node scripts/benchmark-kimi-cache.mjs
   ```
   - 5개 케이스 × 2 라운드 = **10회 호출** 수행
   - 라운드 1 (cache miss 예상) → 5초 대기 → 라운드 2 (cache hit 예상)
   - 결과가 이 파일 아래에 자동으로 append

4. **Cloudflare 대시보드 교차 확인**
   - https://dash.cloudflare.com/ > Workers & Pages > AI > Usage
   - 최근 10회 호출의 **Neurons used** 를 확인
   - 만약 라운드 2가 라운드 1보다 뚜렷하게 적은 neurons를 사용했다면 **prompt caching 동작 확인**

## 판정 기준

- **✅ 통과**: 라운드 2의 `cached_tokens` > 라운드 1, 또는 Cloudflare 대시보드의 neurons가 라운드 2에서 50% 이상 감소
- **⚠️ 부분 통과**: `cached_tokens` 필드는 반환되지 않지만 neurons 사용량이 감소 → 카탈로그 페이지의 `cached input $0.10/M` 표기는 유효
- **❌ 실패**: 라운드 간 차이 없음 → FAQ/가이드 크기 축소하거나 다른 모델(Llama 3.3, DeepSeek) 고려

### 캐시 효과에 대한 관찰

- **지연시간**: Round 1 56.6초 → Round 2 32.8초 — **42.1% 감소** (확인됨)
- **Neurons 차트**: 10개 막대가 25k~50k 범위에서 고르게 분포 — Round 1/2 간 뚜렷한 neurons 감소는 **차트 표시상 보이지 않음**
- **해석 가설 (검증 필요)**:
  - (A) Cloudflare 대시보드의 "Neurons" 집계는 tokens 기반 모델도 neurons으로 환산해 합산 표시. `cached input $0.10/M` 할인이 개별 호출 단위론 반영되나 대시보드 집계엔 단순합으로 보일 가능성
  - (B) tokens-based 모델(Kimi K2.5)에서 caching이 실제로는 지연만 개선하고 청구엔 반영 안 될 수 있음
  - (C) 확정 판정은 **월 청구서(Billing > Invoice)** 확인 시점에만 가능
- **결론**: 지연 개선(42%)은 확실한 운영자 경험 이득이지만, **비용 관점에선 "호출당 약 27원, 월 600건 기준 약 1.6만원"** 을 전제로 예산 편성 권장

### 판정

- ✅ **운영 투입 가능** — 호출당 27원은 고객 지원 문의 하나당 비용으로 수용 가능한 수준
- ⚠️ **캐시 기대는 절반으로** — 지연엔 확실한 효과, 비용엔 증거 불충분

### 답변 샘플 (Round 2 — 각 케이스 첫 300자)

**[q1-group]**
> 현재 번개가입에서는 소셜 가입 시 **사업자회원과 일반회원을 선택하는 기능을 지원하지 않습니다**. 모든 소셜 가입 고객은 쇼핑몰에 설정된 기본 회원등급(현재는 일반회원)으로 자동 등록됩니다.  이는 카페24 SSO 연동 구조상 소셜 로그인 과정에서 추가 정보(사업자번호 등)를 수집할 수 없는 기술적 한계 때문이며, 번개가입 현재 버전에서는 개선 일정이 확정되어 있지 않습니다.  **운영 대안은 다음 두 가지를 권장합니다:**  1. **사업자회원은 기존 가입 폼 유지**: 소셜 가입은 일반회원 전용으로 운영하고, 사업자회원으로 가

**[q2-move]**
> 현재 구글/네이버 로그인 시 표시되는 **"번개가입으로 이동"** 문구는 **변경이 불가능**합니다. 쇼핑몰 개별 브랜딩으로 커스터마이징하는 기능은 현재 지원되지 않습니다.  해당 문구는 고객이 소셜 로그인 버튼을 클릭했을 때 나타나는 **구글/네이버 OAuth 동의 화면**에서 표시되는 앱 이름 때문입니다. 번개가입은 공용 OAuth 앱으로 동작하므로, 모든 쇼핑몰에서 동일하게 "번개가입"이라는 이름으로 노출됩니다.  참고로 말씀하신 **"powered by 번개가입"** 하단 브랜딩은 Plus 요금제에서 숨김 처리가 가능하지만,

**[q3-mobile]**
> PC에서는 정상적으로 소셜 로그인이 되지만 모바일에서만 "유효하지 않은 값" 오류가 발생하는 경우, 카페24 관리자의 **모바일 스킨 SSO 설정**이 별도로 필요하거나 **모바일 도메인 리디렉션 설정** 문제일 가능성이 높습니다.  우선 번개가입 대시보드의 **설정 > SSO 설정 가이드**에 표시된 값(Client ID, Secret, Redirect URL 등)이 카페24 관리자에 정상적으로 등록되어 있는지 다시 한번 확인해 주세요. 특히 카페24의 경우 PC와 모바일 스킨이 분리되어 운영될 수 있어, 모바일 환경에서도 동일한

**[q4-invalid]**
> "유효하지 않은 값" 오류는 설정값 입력 오류일 가능성이 높아 정확한 복사·붙여넣기를 먼저 확인해 주시기 바랍니다. 설정 후 일정 시간이 지나면 자동 적용되지 않으며, 값이 틀린 경우 계속 동일한 오류가 발생합니다.  해당 오류는 주로 카페24 관리자에 입력하신 Client ID, Client Secret, Redirect URL 중 하나가 번개가입 대시보드의 **SSO 설정 가이드**에 표시된 값과 일치하지 않을 때 발생합니다. 특히 Secret 키의 끝 공백이나 URL의 `/` 기호 하나라도 다르면 인증이 실패합니다.  우선 번

**[q5-position]**
> 번개가입 관리자 페이지에서 '위젯 삽입 위치' 설정을 **'로그인 폼 아래'**로 변경하시면 소셜 로그인 섹션이 하단으로 이동합니다. 또는 커스텀 제작 스킨이라면 CSS 셀렉터를 직접 입력하여 원하는 위치에 배치할 수도 있습니다.  **변경 방법은 다음과 같습니다:**  1. 번개가입 관리자 대시보드에 접속하셔서 **'디자인'** 메뉴로 이동합니다. 2.
