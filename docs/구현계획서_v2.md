## Phase 3: 기존 회원-소셜 계정 연결 (1.5주)

### 개요
- 이미 이메일/비밀번호로 가입한 카페24 회원이 소셜 로그인 시도 시 기존 계정과 연결
- 카페24 SSO의 Account Linking 기능과 연동

### 플로우

```
1. 소셜 로그인 → 이메일 수집
2. shop_users에서 email_hash로 동일 이메일 기존 회원 검색
3-A. 기존 회원 없음 → 기존 플로우 (신규 가입)
3-B. 기존 회원 있음 (다른 provider) → Account Linking 플로우
   4. "기존 회원입니다" 중간 페이지 표시
   5-A. 사용자가 "연결" 선택 → 동일 user에 새 provider 매핑
   5-B. 사용자가 "새 계정" 선택 → 별도 계정으로 가입
```

### 구현 체크리스트

#### 3-1. DB 스키마 변경
- [ ] `users` 테이블: 1 user → N provider 지원
  - 현재: `provider` + `provider_uid`가 users 테이블에 직접 있음
  - 변경: `user_providers` 테이블 분리
    ```sql
    CREATE TABLE user_providers (
      id TEXT PRIMARY KEY,
      user_id TEXT NOT NULL REFERENCES users(user_id),
      provider TEXT NOT NULL,
      provider_uid TEXT NOT NULL,
      linked_at TEXT DEFAULT (datetime('now')),
      UNIQUE(provider, provider_uid)
    );
    ```
  - 기존 users.provider, users.provider_uid 데이터를 user_providers로 마이그레이션
- [ ] `docs/schema.sql` 업데이트
- [ ] D1 마이그레이션 실행 + 기존 데이터 이관

#### 3-2. 계정 연결 로직 (oauth.ts 콜백)
- [ ] 콜백에서 email_hash로 기존 user 조회
- [ ] 기존 user 있고, 다른 provider인 경우:
  - KV에 `linking_session:{session_id}` 저장 (신규 provider 정보 + 기존 user_id)
  - Account Linking 중간 페이지로 리다이렉트
- [ ] 기존 user 없는 경우: 기존 플로우 유지

#### 3-3. Account Linking 페이지 + API
- [ ] `GET /oauth/link` — 중간 페이지 렌더링 (SSR)
  - "이미 {기존provider}로 가입된 계정입니다. 연결하시겠습니까?"
  - "연결하기" / "새 계정으로 가입" 버튼
- [ ] `POST /oauth/link/confirm` — 연결 확인
  - user_providers에 새 provider 레코드 추가
  - 기존 user_id로 auth_code 발급
  - 카페24 redirect_uri로 리다이렉트
- [ ] `POST /oauth/link/skip` — 별도 계정 생성
  - 새 user 생성 + 기존 플로우 진행

#### 3-4. queries.ts 확장
- [ ] `getUserByEmailHash(emailHash)` — 이메일 해시로 기존 사용자 조회
- [ ] `getUserProviders(userId)` — 사용자의 연결된 프로바이더 목록
- [ ] `addUserProvider(userId, provider, providerUid)` — 프로바이더 연결 추가
- [ ] `upsertUser` 수정 — user_providers 테이블 사용하도록 변경

## Phase 6: Telegram 프로바이더 추가 (1.5주)

### 개요
- Telegram은 OAuth가 아닌 **Login Widget** 방식
- 위젯이 사용자 데이터를 직접 반환 (HMAC-SHA-256 서명 포함)
- 제공 정보: id, first_name, last_name, username, photo_url (전화번호/이메일 없음)
- 기존 OAuth 플로우에 맞추되 authorize→callback 단계가 다름

### 사전 작업
- [ ] Telegram @BotFather에서 봇 생성 (`/newbot`)
- [ ] `/setdomain`으로 `bg.suparain.kr` 등록
- [ ] Bot Token 획득
- [ ] Wrangler Secret 등록: `TELEGRAM_BOT_TOKEN`

### 아키텍처 차이

```
일반 OAuth:
  위젯 클릭 → /oauth/authorize → 소셜 서비스 → /oauth/callback → auth_code

Telegram:
  위젯 클릭 → Telegram Widget 팝업 → 클라이언트에 데이터 반환
  → POST /oauth/callback/telegram (데이터 + HMAC) → 서버 검증 → auth_code
```
