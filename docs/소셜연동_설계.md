# 소셜 계정 연동 설계서

> 작성일: 2026-03-28
> 상태: Draft

---

## 1. 개요

### 1.1 목적

번개가입의 "소셜 계정 연동" 기능은 카페24 쇼핑몰 기존 회원이 마이페이지에서 추가 소셜 계정을 연결하고, 이후 해당 소셜 계정으로도 **동일한 카페24 회원**으로 로그인할 수 있게 하는 기능이다.

### 1.2 범위

- 마이페이지 연동 위젯 (UI/UX)
- 연동 API (서버 사이드)
- 카페24 member_id 매핑 (핵심)
- 연동 해제 기능
- 연동 상태 조회 API

### 1.3 핵심 가치

| 관점 | 기대 효과 |
|------|-----------|
| 쇼핑몰 고객 | 여러 소셜 계정 중 아무거나로 로그인 가능 |
| 쇼핑몰 운영자 | 중복 회원 감소, 회원 관리 간소화 |
| 번개가입 서비스 | 유저당 연동 계정 수 증가 = lock-in 강화 |

## 2. 현재 구현 상태

### 2.1 소셜 로그인 플로우 (정상 작동 중)

```
[쇼핑몰 위젯] → provider hint (KV) → MemberAction.snsLogin('sso')
    → [카페24] → GET /oauth/authorize (client_id, redirect_uri, state)
    → [번개가입] → KV hint로 provider 선택 → 소셜 OAuth 진행
    → [소셜 프로바이더] → 인증 완료
    → [번개가입] → /oauth/callback/:provider → upsertUser → auth_code 생성
    → [카페24] → redirect(auth_code + state)
    → [카페24] → POST /oauth/token → access_token 교환
    → [카페24] → POST /oauth/userinfo → 사용자 정보 반환
    → [카페24] → 회원 생성/로그인 완료
```

### 2.2 계정 연결 로직 (upsertUser)

현재 upsertUser는 3단계 로직으로 작동한다:

1. **provider + provider_uid** 조회: 동일 소셜 계정이면 기존 유저 업데이트
2. **email_hash** 조회: 다른 프로바이더지만 같은 이메일이면 자동 계정 연결 (user_providers에 추가)
3. **신규 생성**: 둘 다 매칭 안 되면 새 유저 생성

### 2.3 마이페이지 연동 위젯 (UI만 구현)

- 마이페이지 왼쪽 메뉴에 "소셜 계정 연동" 메뉴 삽입
- 클릭 시 모달 팝업으로 연동 관리 UI 표시
- "연동하기" 버튼 클릭 시 팝업 윈도우로 OAuth 플로우 진행
- redirect_uri를 `/link/complete`로 설정하여 팝업 내에서 "연동 완료" 표시 후 닫힘
- 연동 상태는 **localStorage 기반**으로만 추적 (서버 API 미구현)

### 2.4 DB 스키마 현황

관련 테이블:

| 테이블 | 역할 | 비고 |
|--------|------|------|
| `users` | 소셜 유저 마스터 | provider, provider_uid, email_hash 보유 |
| `user_providers` | 유저별 연동 소셜 계정 목록 | user_id, provider, provider_uid |
| `shop_users` | 쇼핑몰-유저 매핑 | **platform_member_id 컬럼 존재하나 미사용** |

`shop_users.platform_member_id`는 스키마 설계 시 카페24 member_id를 저장하기 위해 만들어졌으나, 현재 코드에서 전혀 활용되지 않는다.

## 3. 핵심 문제점

### 3.1 카페24 회원과 소셜 계정의 연결 고리 부재

현재 구조에서 세 가지 식별자가 독립적으로 존재한다:

```
카페24 member_id ←(연결 없음)→ 번개가입 user_id ←(user_providers)→ social provider_uid
```

이로 인해 다음 시나리오가 발생한다:

**시나리오: 마이페이지 연동 후 소셜 로그인 시 중복 회원 생성**

1. 사용자 A가 이메일/비밀번호로 카페24에 직접 회원가입 (member_id: `abc123`)
2. 마이페이지에서 "네이버 연동" 클릭 → 네이버 OAuth 인증 → 번개가입 DB에 기록
3. 다음에 로그인 페이지에서 "네이버로 시작하기" 클릭
4. **기대**: member_id `abc123`으로 로그인
5. **실제**: 카페24 SSO를 통해 새 회원 B가 생성됨 (member_id: `def456`)

**원인**: 마이페이지 연동 시 카페24 member_id가 번개가입 DB에 저장되지 않기 때문에, 이후 SSO 로그인 시 카페24가 userinfo의 `id`를 새 회원으로 취급한다.

### 3.2 SSO userinfo 응답의 구조적 한계

카페24 SSO가 `/oauth/userinfo`로 받는 응답에서 `id` 필드로 회원을 식별한다:

```json
{ "id": "번개가입_user_id", "email": "...", "name": "..." }
```

카페24는 이 `id`가 이전에 한 번도 본 적 없는 값이면 **무조건 새 회원을 생성**한다. 마이페이지 연동이 아무리 잘 되어 있어도, SSO 플로우에서 기존 member_id와의 매핑 정보가 없으면 문제가 해결되지 않는다.

### 3.3 마이페이지 연동의 서버 사이드 부재

현재 마이페이지 위젯은:
- 연동 상태를 localStorage에만 저장 (브라우저/기기 변경 시 유실)
- 서버에 "이 카페24 회원이 이 소셜 계정을 연동했다"는 정보를 전달하지 않음
- 연동 해제 API도 없음

## 4. 목표 아키텍처

### 4.1 핵심 설계 원칙

> **카페24 member_id를 번개가입 user_id에 바인딩하는 것이 이 기능의 핵심이다.**

연동이 완료되면 `shop_users.platform_member_id`에 카페24 member_id가 저장되어야 하고, 이후 SSO 로그인 시 이 매핑을 활용하여 동일 회원으로 로그인되어야 한다.

### 4.2 연동 플로우 (목표 상태)

#### Phase A: 마이페이지에서 소셜 계정 연동

```
[마이페이지 위젯]
  1. CAFE24API.getMemberID() 로 현재 member_id 획득
  2. "네이버 연동" 클릭
  3. 팝업 윈도우 열기: /link/start?client_id=X&provider=naver&member_id=M
     ↓
[번개가입 서버 /link/start]
  4. member_id + client_id + provider를 KV 세션에 저장
  5. 소셜 OAuth 시작 (네이버 로그인 페이지로 리다이렉트)
     ↓
[네이버]
  6. 사용자 인증 완료
     ↓
[번개가입 서버 /link/callback/:provider]
  7. 소셜 userinfo 획득
  8. upsertUser로 번개가입 user 생성/조회
  9. shop_users 테이블에서 (shop_id, user_id) 레코드에 platform_member_id = M 저장
  10. user_providers 테이블에 provider 추가 (이미 upsertUser에서 처리)
  11. /link/complete 페이지 반환 → 팝업 닫힘 → 부모 창에 postMessage로 성공 알림
     ↓
[마이페이지 위젯]
  12. 연동 상태 갱신 (서버 API 조회)
```

#### Phase B: 연동된 소셜 계정으로 로그인

```
[로그인 페이지 위젯]
  1. "네이버로 시작하기" 클릭 → provider hint → MemberAction.snsLogin('sso')
     ↓
[카페24 → 번개가입 /oauth/authorize → /oauth/callback/naver]
  2. 기존 SSO 플로우 그대로 진행
  3. upsertUser로 유저 조회/업데이트
     ↓
[번개가입 /oauth/userinfo (기존 로직 확장)]
  4. shop_users에서 user_id + shop_id로 platform_member_id 조회
  5-a. platform_member_id가 있으면:
       → userinfo 응답에 id = platform_member_id 반환
       → 카페24가 기존 회원으로 인식하여 로그인 처리
  5-b. platform_member_id가 없으면:
       → 기존과 동일하게 id = user_id 반환
       → 카페24가 신규 회원 생성 (기존 동작)
```

### 4.3 데이터 관계도 (목표 상태)

```
카페24 member_id ←(shop_users.platform_member_id)→ 번개가입 user_id
                                                           ↕
                                                    user_providers
                                                     ├─ google
                                                     ├─ naver
                                                     └─ kakao
```

이 구조가 완성되면:
- member_id `abc123` ↔ user_id `bg_xxx` ↔ [google, naver, kakao]
- google/naver/kakao 어느 것으로 로그인해도 카페24에서 member_id `abc123`으로 로그인

### 4.4 userinfo 응답의 id 필드 전략

이 설계의 가장 중요한 결정은 **userinfo 응답의 `id` 필드를 어떤 값으로 반환할 것인가**이다.

| 상황 | id 반환값 | 결과 |
|------|-----------|------|
| platform_member_id가 있음 | platform_member_id (카페24 member_id) | 기존 회원으로 로그인 |
| platform_member_id가 없음 (최초 소셜 로그인) | 번개가입 user_id | 카페24가 신규 회원 생성 |

**주의**: 카페24가 userinfo의 `id`로 회원을 식별하는 정확한 메커니즘을 검증해야 한다. 카페24 SSO 문서에서 `id` 필드의 역할과 기존 회원 매칭 로직을 확인할 필요가 있다.

**대안 검토**:

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. userinfo id에 member_id 반환 | 연동된 경우 카페24 member_id를 id로 반환 | 카페24 측 변경 불필요 | 카페24 SSO가 id 기반으로 기존 회원 매칭을 하는지 검증 필요 |
| B. 카페24 Admin API로 회원 매핑 | `/admin/customersprivacy`로 회원 조회 후 sso_id 업데이트 | 확실한 매핑 | 카페24 Admin API 호출 필요, rate limit 제약, 복잡도 증가 |
| C. 이메일 기반 매칭 (카페24 측) | 동일 이메일이면 카페24가 자동으로 기존 회원에 매핑 | 추가 개발 없음 | 이메일 없는 프로바이더 불가, 카페24의 이메일 매칭 정책에 의존 |

**권장**: 방안 A를 1차로 시도하고, 카페24 SSO의 id 기반 매칭이 작동하지 않는 경우 방안 B로 전환한다.

## 5. 필요한 변경사항

### 5.1 DB 스키마

**변경 없음** -- `shop_users.platform_member_id` 컬럼이 이미 존재한다. 코드에서 활용하기만 하면 된다.

추가 고려사항:
- `shop_users`에 `platform_member_id` 인덱스 추가 (역방향 조회용)
- `user_providers`에 `unlinked_at` 컬럼 추가 고려 (연동 해제 이력 관리)

### 5.2 API 엔드포인트 (신규)

| 메서드 | 경로 | 역할 |
|--------|------|------|
| GET | `/link/start` | 마이페이지 연동 플로우 시작 (member_id 수신, 소셜 OAuth 시작) |
| GET | `/link/callback/:provider` | 소셜 인증 완료 후 연동 처리 (member_id 매핑 저장) |
| GET | `/link/complete` | 연동 완료 UI (팝업 닫힘 + postMessage) -- 기존 엔드포인트 개선 |
| GET | `/api/link/status` | 현재 회원의 연동 상태 조회 (어떤 소셜 계정이 연동되어 있는지) |
| POST | `/api/link/unlink` | 특정 소셜 계정 연동 해제 |

### 5.3 기존 API 변경

**`/oauth/userinfo` 변경**:
- 기존: `id` 필드에 항상 `user_id` 반환
- 변경: `shop_users.platform_member_id`가 있으면 해당 값을 `id`로 반환

**`/oauth/callback/:provider` 변경 (선택)**:
- 최초 소셜 로그인 시 카페24가 생성한 member_id를 역으로 받아와 저장하는 로직은, 카페24 SSO의 콜백 구조상 불가능할 수 있음
- 이 경우 최초 소셜 로그인 유저는 연동되지 않은 상태로 남고, 마이페이지에서 직접 연동해야 함

### 5.4 위젯 변경

**마이페이지 연동 위젯 개선**:
- `CAFE24API.getMemberID()`로 현재 member_id 획득
- 연동 시작 URL에 member_id 포함 (`/link/start?...&member_id=M`)
- localStorage 대신 서버 API (`/api/link/status`)로 연동 상태 조회
- 연동 해제 버튼 추가 → `/api/link/unlink` 호출
- 팝업 완료 시 postMessage 기반으로 부모 창 갱신

**인증 방식**:
- 마이페이지 API 호출 시 인증이 필요한데, 현재 위젯은 카페24 프론트엔드에서 실행됨
- `CAFE24API.getMemberID()`로 얻은 member_id를 신뢰할 수 있는지 검증 필요
- 대안: 위젯이 번개가입에 직접 인증하는 대신, member_id + shop의 client_id + HMAC 서명으로 요청 무결성 검증

### 5.5 OAuth 플로우 분리

현재 `/oauth/authorize`가 "로그인용 SSO 플로우"와 "마이페이지 연동 플로우"를 모두 처리하고 있다. 이를 분리해야 한다:

| 플로우 | 진입점 | redirect_uri | 목적 |
|--------|--------|-------------|------|
| SSO 로그인 | `/oauth/authorize` | 카페24 SSO 콜백 URL | 카페24에 auth_code 전달 → 회원 로그인/생성 |
| 마이페이지 연동 | `/link/start` | `/link/callback/:provider` | 소셜 계정을 기존 카페24 회원에 바인딩 |

두 플로우의 차이:
- **SSO 로그인**: auth_code → token → userinfo → 카페24가 회원 처리
- **마이페이지 연동**: 소셜 인증 → member_id 매핑 저장 → 팝업 닫힘 (카페24 SSO 미개입)

## 6. 고려사항 및 제약

### 6.1 Apple 릴레이 이메일

Apple은 "Hide My Email" 옵션 시 `xxxxx@privaterelay.apple.com` 형태의 릴레이 이메일을 제공한다.

- 이 이메일은 Apple 앱별로 고유하므로, 다른 프로바이더의 실제 이메일과 매칭되지 않음
- email_hash 기반 자동 계정 연결이 불가능
- **해결**: Apple 사용자는 마이페이지에서 명시적으로 연동해야 하며, 이것이 member_id 매핑의 가치가 더욱 높아지는 이유

### 6.2 이메일 없는 프로바이더

| 프로바이더 | 이메일 제공 | 비고 |
|-----------|-----------|------|
| Google | O | 항상 제공 |
| Kakao | O | 동의 필요 |
| Naver | O | 항상 제공 |
| Apple | 조건부 | 릴레이 이메일 또는 미제공 |
| Discord | O | 동의 필요 |
| Telegram | X | 이메일 미제공 |

이메일이 없는 프로바이더(Telegram 등)는:
- email_hash 기반 자동 계정 연결 불가
- 마이페이지 연동(member_id 매핑)이 유일한 기존 회원 연결 방법
- 현재 대체 이메일 생성 로직(`name@shophost`)이 있으나, 이는 카페24 SSO 필수 필드 충족용일 뿐 계정 연결에는 사용 불가

### 6.3 member_id 신뢰성

마이페이지 위젯에서 `CAFE24API.getMemberID()`로 얻는 member_id의 신뢰성:

- **위험**: 클라이언트 사이드에서 member_id를 조작할 수 있음
- **완화 방안**:
  - `common_member_id_crypt` (암호화된 member_id)를 사용하여 서버 사이드에서 검증
  - 카페24 Admin API로 해당 member_id의 존재 여부 검증
  - `/link/start` 호출 시 Referer 검증 (쇼핑몰 도메인에서만 허용)
  - rate limiting으로 무차별 연동 시도 방지

### 6.4 카페24 SSO id 필드의 동작 검증

가장 큰 불확실성은 **카페24 SSO가 userinfo의 `id` 값으로 기존 회원을 어떻게 매칭하는지**이다:

- 카페24가 `id` 값을 내부적으로 어디에 저장하는지
- 이전에 `id: "user_abc"`로 생성된 회원에게 다음 로그인에서 같은 `id`가 오면 자동으로 매칭하는지
- `id` 값이 변경되면 (user_id → member_id) 어떤 일이 발생하는지

**검증 방법**: 테스트 쇼핑몰에서 동일 `id`로 두 번 SSO 로그인하여 동일 회원으로 로그인되는지 확인한다.

### 6.5 기존 유저 마이그레이션

이미 소셜 로그인으로 가입한 기존 유저들은 `shop_users.platform_member_id`가 NULL이다:

- 이 유저들이 다시 같은 소셜 계정으로 로그인하면 동일 user_id가 반환되므로 카페24에서도 동일 회원으로 인식됨 (문제 없음)
- 다만 마이페이지 연동 기능을 사용하려면 최초 1회 member_id 바인딩이 필요
- 기존 유저에 대한 자동 바인딩은 불가능 (카페24 member_id를 알 수 없으므로)

### 6.6 동시성 및 엣지 케이스

- **중복 연동 방지**: 이미 다른 카페24 회원에 연동된 소셜 계정을 다시 연동하려는 경우 에러 반환
- **1:N 관계**: 하나의 소셜 계정은 하나의 user_id에만 연결 가능 (user_providers UNIQUE 제약)
- **N:1 관계**: 하나의 user_id에 여러 소셜 계정 연결 가능
- **cross-shop 연동**: 동일 소셜 계정이 여러 쇼핑몰에서 각각 다른 member_id에 연동될 수 있음 (shop_users가 shop_id별로 분리되어 있으므로 정상)

## 7. 구현 우선순위

### Phase 1: 핵심 매핑 인프라 (필수, 최우선)

1. **카페24 SSO id 필드 동작 검증**
   - 테스트 쇼핑몰에서 동일 id로 반복 SSO 로그인 테스트
   - id 변경 시 카페24 동작 확인
   - 이 검증 결과에 따라 방안 A vs B 결정

2. **`/link/start` + `/link/callback/:provider` 엔드포인트 구현**
   - member_id를 KV 세션에 저장하는 연동 시작 플로우
   - 소셜 인증 완료 후 `shop_users.platform_member_id`에 member_id 저장

3. **`/oauth/userinfo` 수정**
   - platform_member_id가 있으면 해당 값을 id로 반환

4. **위젯에서 `CAFE24API.getMemberID()` 연동**
   - 마이페이지 연동 시 member_id를 서버에 전달

### Phase 2: 연동 관리 API (사용성)

5. **`/api/link/status` 연동 상태 조회 API**
   - 현재 member_id에 연동된 소셜 계정 목록 반환
   - 위젯이 localStorage 대신 이 API로 상태 표시

6. **`/api/link/unlink` 연동 해제 API**
   - 특정 소셜 계정의 연동 해제 (user_providers 삭제 또는 soft delete)
   - 최소 1개 연동은 유지하도록 제한 (모든 연동 해제 방지)

7. **위젯 UI 개선**
   - 서버 API 기반 연동 상태 표시
   - 연동 해제 버튼 추가
   - postMessage 기반 팝업-부모 창 통신

### Phase 3: 보안 강화 (안정성)

8. **member_id 검증 강화**
   - `common_member_id_crypt` 활용 또는 카페24 Admin API 검증
   - Referer 검증 + rate limiting

9. **`shop_users.platform_member_id` 인덱스 추가**
   - 역방향 조회 성능 확보

10. **감사 로그**
    - 연동/해제 이벤트를 audit_logs에 기록

### Phase 4: 고도화 (선택)

11. **자동 member_id 바인딩**
    - SSO 로그인 플로우에서 카페24가 생성한 member_id를 역으로 획득하여 자동 저장
    - 카페24 Admin API 연동 필요

12. **기존 유저 마이그레이션 도구**
    - 관리자 앱에서 미연동 유저 목록 확인 및 수동 매핑

13. **연동 알림**
    - 연동 완료/해제 시 사용자에게 알림 (쇼핑몰 알림 또는 이메일)
