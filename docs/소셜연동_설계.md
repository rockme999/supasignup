## 1. 개요

### 1.1 목적

번개가입의 "소셜 계정 연동" 기능은 번개가입으로 소셜 로그인한 카페24 회원(@s)이 마이페이지에서 추가 소셜 계정을 연결하고, 이후 어떤 소셜 계정으로 로그인하든 **동일한 @s 회원**으로 로그인할 수 있게 하는 기능이다.

### 1.2 범위 (수정됨)

- 마이페이지 연동 위젯 (UI/UX)
- @s 회원 내 멀티 소셜 계정 관리
- 연동 상태 조회 / 해제 API
- 서비스 전환(경쟁사→번개가입) 가이드

### 1.3 핵심 가치

| 관점 | 기대 효과 |
|------|-----------|
| 쇼핑몰 고객 | 여러 소셜 계정 중 아무거나로 로그인 가능 |
| 쇼핑몰 운영자 | 소셜 계정 간 중복 @s 회원 감소 |
| 번개가입 서비스 | 유저당 연동 계정 수 증가 = lock-in 강화 |

### 1.4 범위 수정 (2026-03-31 업데이트)

> ~~**ID/PW 기존 회원과 @s 회원의 통합은 이 기능의 범위에 포함되지 않는다.**~~
> ~~카페24 플랫폼의 근본적 제약으로 불가능하다. (섹션 3 참조)~~
>
> **[2026-03-31 수정]** 카페24 "회원가입 시 SNS 계정 연동" 기본 기능으로 해결 가능함이 실테스트로 확인됨.
> 동일 이메일 기반으로 기존 ID/PW 회원과 소셜 계정 연동이 가능하다. (섹션 3.6 참조)

## 2. 카페24 SSO 매칭 메커니즘 (실테스트 결과)

> **2026-03-29 테스트 쇼핑몰(suparain999.cafe24.com)에서 실증 완료**

### 2.1 테스트 결과 요약

| # | 테스트 조건 | 결과 |
|---|------------|------|
| 1 | userinfo `id`를 완전히 다른 값으로 변경 + 이메일 동일 | 기존 @s 회원으로 로그인 (새 회원 안 생김) |
| 2 | ID/PW 회원의 이메일을 SSO 이메일과 동일하게 변경 후 SSO 로그인 | ID/PW 회원 매칭 안 됨, 기존 @s 회원으로 로그인 |

### 2.2 확인된 카페24 SSO 매칭 로직

```
카페24가 /oauth/userinfo 응답을 받으면:

  1. email로 기존 @s 회원 검색
  2. 동일 이메일 @s 회원 있음 → 그 회원으로 로그인
  3. 동일 이메일 @s 회원 없음 → 새 @s 회원 생성 (숫자@s 형식)

  ★ id 필드는 매칭에 사용되지 않음
  ★ @s 회원끼리만 매칭됨 (ID/PW, @g, @n, @k 회원과 교차 매칭 안 됨)
```

### 2.3 카페24 회원 ID 생성 규칙

| 가입 경로 | ID 형식 | 매칭 범위 |
|-----------|---------|-----------|
| ID/PW 직접 가입 | 사용자 지정 (`happyuser`) | ID/PW끼리만 |
| 카페24 구글 | `숫자@g` | @g끼리만 |
| 카페24 네이버 | `숫자@n` | @n끼리만 |
| 카페24 카카오 | `숫자@k` | @k끼리만 |
| 카페24 애플 | `숫자@a` | @a끼리만 |
| SSO 서비스 1번 슬롯 | `숫자@s` | @s끼리만 |
| SSO 서비스 2번 슬롯 | `숫자@s1` | @s1끼리만 |

### 2.4 이메일 매칭의 한계

이메일 기반 매칭이므로 프로바이더별 이메일 제공 여부가 중요하다:

| 프로바이더 | 이메일 제공 | 매칭 신뢰도 |
|-----------|-----------|------------|
| Google | 항상 제공 | ★★★ 확실 |
| Naver | 동의 시 제공 | ★★ 대체로 가능 |
| Kakao | 미제공 가능 | ★ 불안정 |
| Apple | 릴레이 이메일 or 미제공 | ★ 불안정 |
| Discord | 동의 시 제공 | ★★ 대체로 가능 |
| Telegram | 미제공 | ✗ 매칭 불가 |

이메일이 없는 프로바이더는 대체 이메일(`name@shophost`)을 생성하는데, 이름이나 호스트가 바뀌면 카페24가 다른 회원으로 인식할 수 있다.

→ **이메일을 유니크한 회원 식별 기준으로 삼는 것은 불가능하다.**

## 3. 카페24 플랫폼의 연동 메커니즘

### 3.1 ID/PW 회원과 @s 회원은 통합 불가 → [2026-03-31 수정: 조건부 가능]

**[2026-03-31 업데이트]** 아래 기존 내용은 "카페24 기본 SNS 연동 설정 미사용 시"의 동작이다. 설정 활성화 시 동작이 달라짐 (섹션 3.6 참조).

```
[기존 동작 — "회원가입 시 SNS 계정 연동" 설정 OFF일 때]
ID/PW 회원 (happyuser, email: a@b.com)
        ↕ 카페24 내부에서 연결 안 됨
@s 회원 (2784646035@s, email: a@b.com)

→ 같은 이메일이어도 카페24는 별개 회원으로 관리
→ 마일리지, 쿠폰, 주문내역, 회원등급 각각 별도
```

### 3.2 회원 병합 API 없음, 하지만 혜택 이전은 부분적으로 가능

카페24 Admin API에 회원 병합(merge) API는 존재하지 않는다. 하지만 개별 API를 조합하면 혜택 이전이 부분적으로 가능하다:

| 기능 | API | 가능 여부 |
|------|-----|:---------:|
| 적립금 조회 | `GET /api/v2/admin/points` | ✅ |
| 적립금 적립/차감 | `POST /api/v2/admin/points` (scope: `mall.write_mileage`) | ✅ |
| 쿠폰 목록 조회 | `GET /api/v2/admin/customers/{member_id}/coupons` | ✅ |
| 쿠폰 발급 | `POST /api/v2/admin/coupons/issues` (scope: `mall.write_promotion`) | ✅ |
| 쿠폰 삭제 | `DELETE /api/v2/admin/customers/{member_id}/coupons/{coupon_id}` | ✅ |
| 주문내역 이전 | - | ❌ 불가 |
| 회원등급 변경 | - | ❌ 확인 필요 |

**혜택 이전 시나리오 (ID/PW → @s):**
```
1. 번개가입 DB에서 user1 ↔ 123456@s 매핑을 알고 있음 (마이페이지 연동으로 획득)
2. Admin API로 user1의 적립금 조회 → 123456@s에 동일 금액 적립 → user1에서 차감
3. Admin API로 user1의 쿠폰 조회 → 123456@s에 동일 쿠폰 발급 → user1의 쿠폰 삭제
4. 주문내역은 이전 불가 (읽기만 가능)
```

**주의사항:**
- 두 API 호출 사이 트랜잭션 보장 안 됨 (중간 실패 시 데이터 불일치 위험)
- 현재 번개가입 앱의 scope에 `mall.write_mileage`, `mall.write_promotion` 미포함 (추가 필요)
- 쿠폰은 동일 쿠폰을 "재발급"하는 것이므로 쿠폰 마스터(coupon_no)가 필요

### 3.3 회원 SNS 연동 정보 조회 API (제한적)

```
GET /api/v2/admin/customers/{member_id}/social
scope: mall.read_customer

응답 예시:
{
    "social": {
        "shop_no": 1,
        "member_id": "sampleid",
        "social_name": "line",
        "social_member_code": "U1e0014229a08c2f95e12ee29904da597",
        "linked_date": "2019-02-18T13:03:11+09:00"
    }
}
```

- `social_member_code`는 SSO의 경우 번개가입이 보낸 userinfo의 `id` 값일 가능성 높음
- **읽기 전용** (GET만 있음, PUT/POST 없음) → 쓰기가 가능하면 ID/PW 회원에 SSO 연동을 직접 주입할 수 있으나 현재 불가
- **접근 제한**: "특정 클라이언트만 사용할 수 있는 API" — 카페24 개발자센터에 사용 신청 필요
- 이 API로 기존 회원의 SSO 연동 상태를 파악하여 중복 여부 판단에 활용 가능

### 3.4 부-자 아이디는 무관

카페24의 "부-자 아이디"는 **쇼핑몰 호스팅 관리자** 계정의 그룹화 시스템이다. 쇼핑몰 **고객** 회원의 계정 통합 기능이 아니므로 이 목적에는 사용 불가.

### 3.4 SNS 로그인 설정 변경 제약

> "카카오를 제외한 다른 SNS 로그인은 이미 가입(연동)한 회원이 있으면 연동 정보나 사용여부를 변경할 수 없어요."

- @g/@n 회원이 존재하면 해당 소셜 로그인 설정을 끌 수 없음
- 단, SSO 서비스는 **2개 동시 등록 가능** (@s + @s1)

### 3.5 경쟁사도 동일한 한계

이 제약은 카페24 플랫폼 자체의 한계이므로 1초가입 등 경쟁 서비스도 동일한 제약을 가진다. 번개가입만의 문제가 아님.

### 3.6 해결됨: "회원가입 시 SNS 계정 연동" 기능 (2026-03-31 실테스트 확인)

카페24가 기본 제공하는 "회원가입 시 SNS 계정 연동" 설정으로 ID/PW 회원과 소셜 계정 연동이 가능함을 실테스트로 확인.

**활성화 경로:** 카페24 관리자 → 회원 설정 → "회원가입 시 SNS 계정 연동" ON

**동작 플로우:**
```
1. 카페24 관리자에서 "회원가입 시 SNS 계정 연동" 활성화
2. 기존 ID/PW 회원과 동일한 이메일로 SSO 로그인 시도
3. 카페24가 "아래 계정과 연동하시겠습니까?" 화면 표시 (기존 계정 목록)
4. 고객이 선택하여 연동 완료
5. 이후 소셜 로그인 시 기존 ID/PW 회원으로 바로 로그인됨
```

**실테스트 결과 (2026-03-31):**
- bbbig(ID/PW 회원)에 SSO 소셜 계정 연동 성공
- Social API 조회: `social_name: "sso"` 연동 레코드 생성 확인
- 연동 후 소셜 로그인 → bbbig으로 바로 로그인 (새 @s 생성 없음)

**핵심 조건:** 이메일이 동일해야 연동 화면이 표시됨

> ⚠️ **중요 전제 조건 — 연동 기능의 동작 범위:**
>
> 이미 해당 소셜 계정으로 SSO를 통해 @s 회원이 생성되어 있다면, 동일 이메일이더라도 연동 화면이 뜨지 않는다. 카페24가 "신규 가입"이 아닌 "기존 @s 회원 로그인"으로 처리하기 때문이다.
>
> 즉, 연동 기능은 **해당 소셜 계정으로 처음 SSO 가입할 때만** 동작한다.
>
> 따라서 운영 시:
> - 쇼핑몰에 번개가입 앱 설치 시 "회원가입 시 SNS 계정 연동" 설정을 **먼저** 활성화해야 함
> - 이미 @s 회원이 생성된 후에는 연동 불가 — 해당 @s 회원을 탈퇴 처리하더라도 카페24에서 재가입이 거부됨

> ⚠️ **위험 — 소셜 계정 탈퇴 시 동일 이메일의 모든 SSO 로그인 차단:**
>
> 카페24는 이메일 기준으로 탈퇴 이력을 관리한다. 동일 이메일을 사용하는 소셜 계정(@s) 중 하나가 탈퇴되면, 해당 이메일을 사용하는 다른 소셜 계정으로의 SSO 로그인 및 신규 가입도 "탈퇴한 계정"으로 차단된다.
>
> **예시:**
> - aaa@aaa.com 이메일의 ID/PW 계정(aaa)과 구글 @s 계정이 있음 (연동되지 않은 상태)
> - 구글 @s 계정을 회원 탈퇴
> - 이후 aaa@aaa.com 이메일의 카카오, 네이버 등 다른 소셜로 SSO 로그인/가입 시도 → "탈퇴한 계정" 메시지로 차단됨
> - ID/PW 계정(aaa)은 별도로 존재하므로 영향 없음
>
> **운영 시 주의사항:**
> - 쇼핑몰 운영자에게 소셜 회원(@s) 탈퇴 시 해당 이메일의 모든 SSO 로그인이 차단됨을 안내해야 함
> - "회원가입 시 SNS 계정 연동" 기능 활성화로 ID/PW 회원과 소셜 계정을 미리 연동해두면 이 문제를 예방할 수 있음 (소셜 탈퇴 대신 연동 해제로 처리)

**프로바이더별 연동 가능 여부:**

| 프로바이더 | 연동 가능 | 이유 |
|-----------|:--------:|------|
| Google | ✅ | 실제 이메일 항상 반환 |
| Kakao | ✅ | 실제 이메일 반환 (동의 시) |
| Naver | ✅ | 실제 이메일 반환 (동의 시) |
| Facebook | ✅ | 실제 이메일 반환 (동의 시) |
| Discord | ✅ | 실제 이메일 반환 |
| LINE | ✅ | 실제 이메일 반환 (동의 시) |
| Apple | ⚠️ | Hide My Email 기본 선택 → 릴레이 이메일로 매칭 불가 가능성 높음 |
| X(Twitter) | ❌ | 이메일 미제공 (Basic 플랜 $100/월 필요) |
| Telegram | ❌ | 이메일 미제공 |

**제한사항:**
- Apple: 사용자가 "이메일 숨기기(Hide My Email)"를 선택하면 릴레이 이메일이 반환되어 기존 계정과 이메일 불일치 → 연동 불가
- X/Telegram: 이메일 자체를 제공하지 않으므로 대체 이메일 생성 → 기존 계정과 매칭 불가

**결론:** 카페24 별도 API 구현 불필요. 쇼핑몰 운영자 설정 가이드만 제공하면 됨.

## 4. 번개가입의 실제 가치 (재정의)

### 4.1 핵심 가치

```
✅ 신규 가입 이탈률 감소
   → 소셜 로그인 1-클릭으로 가입+로그인 완료
   → 기존 회원가입 폼 대비 전환율 대폭 향상

✅ 멀티 프로바이더 지원 (9종)
   → Google, Kakao, Naver, Apple, Discord, Facebook, X, LINE, Telegram
   → 카페24 자체 소셜(4종)보다 넓은 커버리지

✅ @s 회원 내 멀티 소셜 로그인
   → 같은 이메일의 소셜 계정은 카페24가 자동으로 같은 @s 회원에 매칭
   → 번개가입 DB(user_providers)에서도 멀티 소셜 관리

✅ 위젯 커스터마이징
   → 쇼핑몰 브랜딩에 맞는 소셜 로그인 UI 제공
   → 프리셋, 색상, 크기, 배치 등 커스터마이징

✅ 대시보드 분석
   → 프로바이더별 가입/로그인 통계
   → 쇼핑몰 운영자 인사이트 제공
```

### 4.2 명시적 한계 (투명하게 안내)

```
✅ [2026-03-31 수정] 기존 ID/PW 회원과의 소셜 계정 연동
   → 카페24 "회원가입 시 SNS 계정 연동" 설정 활성화 시 가능
```

## 5. 마이페이지 소셜 연동 (수정된 설계)

### 5.1 목적 (수정됨)

~~기존: ID/PW 회원과 소셜 계정을 연결하여 같은 카페24 회원으로 로그인~~

**수정: @s 회원이 자신에게 연결된 소셜 계정 목록을 확인하고 관리하는 기능**

이메일이 같은 소셜 계정은 카페24가 자동 매칭하므로, 마이페이지 연동의 실질적 역할은:
1. 현재 연동된 소셜 계정 **현황 확인** (어떤 소셜로 로그인 가능한지)
2. 새로운 소셜 계정 **추가 연동** (번개가입 DB에 user_providers 추가)
3. 연동 **해제** (특정 소셜 계정 연결 끊기)

### 5.2 현재 구현 상태

- 마이페이지 왼쪽 메뉴에 "소셜 계정 연동" 메뉴 삽입 ✅
- 클릭 시 모달 팝업으로 연동 관리 UI 표시 ✅
- "연동하기" 버튼 클릭 시 팝업 윈도우로 OAuth 플로우 진행 ✅
- 연동 완료 시 `/link/complete` 페이지 표시 ✅
- 연동 상태는 localStorage 기반으로만 추적 (서버 API 미구현) ⚠️

### 5.3 추가 구현 필요 항목

| 메서드 | 경로 | 역할 | 상태 |
|--------|------|------|------|
| GET | `/link/start` | 소셜 연동 플로우 시작 (소셜 OAuth 시작) | 미구현 |
| GET | `/link/callback/:provider` | 소셜 인증 완료 후 user_providers에 추가 | 미구현 |
| GET | `/link/complete` | 연동 완료 UI (팝업 닫힘 + postMessage) | 부분 구현 |
| GET | `/api/link/status` | 현재 회원의 연동 상태 조회 | 미구현 |
| POST | `/api/link/unlink` | 특정 소셜 계정 연동 해제 | 미구현 |

### 5.4 연동 플로우

```
[마이페이지 위젯]
  1. "네이버 연동" 클릭
  2. 팝업: /link/start?client_id=X&provider=naver
     ↓
[번개가입 /link/start]
  3. client_id 검증 → 소셜 OAuth 시작
     ↓
[네이버 인증 완료 → /link/callback/naver]
  4. 소셜 userinfo 획득
  5. upsertUser() → 기존 user 찾거나 신규 생성
  6. user_providers에 naver 추가
  7. /link/complete → 팝업 닫힘 → 부모 창에 postMessage
     ↓
[마이페이지 위젯]
  8. 연동 상태 갱신 (/api/link/status 호출)
```

### 5.5 연동 상태 조회 API

```
GET /api/link/status?client_id=X&user_id=Y

응답:
{
  "providers": [
    { "provider": "google", "linked_at": "2026-03-28T13:27:45Z" },
    { "provider": "naver", "linked_at": "2026-03-29T10:15:00Z" }
  ]
}
```

### 5.6 연동 해제 API

```
POST /api/link/unlink
Body: { "client_id": "X", "user_id": "Y", "provider": "naver" }

제약:
- 최소 1개 소셜 계정은 유지 (전부 해제 불가)
- 해제 후에도 같은 이메일이면 카페24에서는 여전히 같은 @s 회원으로 매칭됨
  (카페24의 매칭은 이메일 기반이므로 번개가입 DB 연동 해제와 무관)
```

### 5.7 platform_member_id 필드 처리

`shop_users.platform_member_id` 컬럼은 원래 카페24 member_id를 저장하여 userinfo의 id 필드를 오버라이드하려는 목적이었으나, **실테스트로 이 전략이 무효화**되었다.

**결정: 당분간 미사용 상태 유지.** 추후 카페24 Admin API 연동이 필요한 경우(회원 정보 조회 등)에 활용할 수 있으므로 컬럼 자체는 삭제하지 않는다.

**설계 변화 (2025-04 이후):** 카페24 `member_id`의 영구 저장은 `shop_users.platform_member_id` 대신 별도 `cafe24_members` 테이블로 처리한다. `member_joined` 웹훅(event 90083) 수신 시 upsert로 저장하며, 쿠폰 발급 외에도 회원 관리·분석 등 향후 기능 확장에 활용하기 위함이다. `shop_users.platform_member_id`는 현재도 미사용 상태이며 별도 설계 결정 전까지 유지만 한다.

### 6.1 전환 대상 유형

### 6.2 유형 A: 소셜 없는 쇼핑몰

```
1. 번개가입 앱 설치 + SSO 설정
2. 위젯 삽입
3. 완료

기존 ID/PW 회원: 기존 방식 로그인 유지

```

### 6.3 유형 B: 카페24 자체 소셜 사용 쇼핑몰

```
제약: @g/@n 회원이 있으면 해당 소셜 설정을 끌 수 없음

전략: "공존"
1. 번개가입 SSO 추가 활성화
2. 번개가입 위젯을 로그인 페이지에 삽입
3. 카페24 자체 소셜 버튼은 스킨 편집으로 CSS 숨김 처리
   (설정은 켜진 상태이나 UI에서 안 보이게)

결과:
- 기존 @g/@n/@k 회원: 로그인 버튼이 숨겨져서 소셜 로그인 불가
  → ID/PW가 있으면 ID/PW로 로그인
  → 소셜 전용 회원은 비밀번호 찾기/재설정 필요
  → 또는 번개가입으로 같은 이메일 소셜 로그인 시 새 @s 회원 생성
- 신규 회원: 번개가입으로 가입 (@s)

주의: 소셜 전용(@g/@n/@k) 회원이 많은 쇼핑몰은
  전환 전 비밀번호 설정 안내 캠페인 필요
```

### 6.4 유형 C: 경쟁 SSO → 번개가입

```
핵심 발견: SSO 2개 동시 등록 가능 (@s + @s1)

전략: "공존 → 점진적 전환"

Phase 1: 공존
  1. SSO 2번 슬롯에 번개가입 추가 (@s1 회원 생성)
  2. 기존 SSO(1초가입) 버튼은 스킨에서 CSS 숨김
  3. 번개가입 위젯만 노출

Phase 2: 이메일 매칭에 의한 자연 전환
  - 기존 @s 회원이 번개가입으로 같은 소셜 로그인 시
  - 이메일이 같다면? → 카페24가... @s1 회원으로 생성 (별도 네임스페이스)
  - 이메일이 같아도 @s와 @s1은 별개 회원!

Phase 3: 기존 @s 회원 처리
  - 1초가입 버튼이 숨겨져 있으므로 @s 회원은 로그인 경로 없음
  - 비밀번호가 있으면 ID/PW 로그인... 하지만 @s 회원에 비밀번호가 있을 가능성 낮음
  - ★ 이것이 가장 큰 문제: 기존 @s 회원 로그인 불가 위험

해법:
  A. 전환 기간 동안 기존 SSO 버튼도 노출 (버튼 2세트)
  B. 기존 @s 회원에게 사전 안내 + 비밀번호 설정 유도
  C. 1초가입 계약 해지 전까지는 공존 유지
```

## 7. 이메일 없는 프로바이더 대응

### 7.1 현재 대체 이메일 생성 로직

```typescript
// 이메일이 없는 프로바이더(Telegram, X 등)
const nameSlug = (name ?? user.provider).replace(/\s+/g, '_');
finalEmail = `${nameSlug}@${shopHost}`;
// 예: "telegram_user@suparain.kr"
```

### 7.2 문제점

- 사용자 이름이 바뀌면 대체 이메일도 바뀜 → 카페24가 다른 회원으로 인식
- 같은 이름의 다른 사용자가 같은 대체 이메일을 가질 수 있음

### 7.3 개선 방안

```
대체 이메일에 provider_uid를 포함하여 고유성 + 일관성 보장:

현재: telegram_user@suparain.kr (이름 기반, 불안정)
개선: telegram_8266607489@suparain.kr (provider_uid 기반, 고유+불변)

형식: {provider}_{provider_uid}@{shopHost}
```

이렇게 하면:
- 같은 Telegram 사용자는 매번 같은 대체 이메일 → 같은 @s 회원 매칭
- 다른 Telegram 사용자는 다른 대체 이메일 → 다른 @s 회원 생성
- provider_uid는 불변이므로 이름 변경에 영향 없음

8. **마이페이지 위젯에 "혜택 이전" UI**

### Phase 4: 서비스 전환 도구 (운영)

10. **기존 회원 안내 템플릿**
